## 3. dict 회사별명칭 ↔ bycover 불일치 3건 정리 (정합 방향)

정합 원칙: **bycover = 소식지 확인 실데이터(진실 원천), dict 회사별명칭 = 명칭 차이 설명(보조).** 정합은 확인된 근거끼리 additive로만. 원본 삭제 없음.

| 담보 | 불일치 | 자료 근거 | 정합 방향(정리본 반영) |
|---|---|---|---|
| 납입면제 | dict={KB손보·한화손보·DB손보} / bycover={KB라이프·한화손보} | KB라이프 "딱좋은 0540 · 11대질병 납입면제" = bycover O(소식지 확인) | **dict에 KB라이프 추가**(O 근거). KB손보·DB손보는 명칭차이 설명으로 유지(bycover 미확인이나 삭제 안 함) |
| 치아치료비 | dict 3사 / bycover 2사 (라이나손보 누락) | 라이나손보 "더핏 THE든든한 치아"는 라인업(GOJI-0114)·v1 246 feature("크라운 40만")에 실재 | **bycover에 라이나손보 △ 추가**(라인업 근거, 금액=크라운 40만). 소식지 O 미확정이라 O 아닌 △ |
| 실손의료비 | dict 2사 / bycover 1사 (KB손보 누락) | KB손보 "5세대 실손"은 dict 트렌드 설명뿐, 소식지 상품·금액 원문 근거 없음 | **bycover 무변경(메리츠만 유지)**. dict KB손보 설명 유지. 원문(KB손보 실손 소식지) 확보 시 추가 — 감사 대기 항목 |

- 방향 요지: 확인 근거 있는 것만 반영(납입면제 KB라이프, 치아 라이나손보). 근거 없는 것(실손 KB손보)은 추정으로 채우지 않고 원문 대기.

## 4. 금액·조건 재확인 (추정 금지)

| 점검 | 결과 |
|---|---|
| 추정 금액 유입 | **0건** (모든 금액이 소식지/라인업 원문 문구) |
| 금액 강등 필요 | **0건** |
| 금액 필드가 액수가 아닌 담보·조건 문구인 항목 | 18건 (예: 실손 "표준/유병력자/계약전환/노후", 3대 "심장진단비+I49진단비", 납입면제 "8대납면") |

- 18건은 "추정 숫자"가 아니라 원문 담보·조건 표기다(추정 금지 원칙 위반 아님). 다만 화면에서 액수로 오해되지 않도록, 액수 없는 항목은 표시 로직에서 담보/조건 문구로 노출(v1 HTML은 이미 액수 없으면 "금액 원문 확인 필요" 분기 보유). 강등 대상 아님.
- 신규 5담보 금액도 전부 원문(간병인사용일당 "최대 13% 인하 대상", 경영인 "해약환급금일부지급형 2601" 등). 추정 없음.

## 정합 반영 요약 (정리본 dambo-cover-v3.json)

1. 납입면제 dict 회사별명칭 += KB라이프(bycover O 근거). KB손보·DB손보 명칭 유지(삭제 0).
2. 치아치료비 bycover += 라이나손해보험 △(라인업 근거, 금액 "크라운 40만 (라인업 근거)").
3. bycover 키 순서를 dict 순서(담보 성격순)로 통일.
4. 실손 KB손보는 원문 대기(추정 미채움) — meta.미반영_원문대기에 기록.
5. 원본 삭제 0. dictionary·colors 내용 무변경(납입면제 명칭 추가 제외).

## 추가 확인 필요 원문 (원문 확보 시 승격)

- KB손해보험 실손 소식지 상품·금액 → bycover 실손 추가 근거
- 간병인사용일당/지원일당 일당 액수(원) → O 금액 확정
- 경영인정기 5건 손금처리·환급 조건 원문 → △ 세분 또는 O 승격
- 치아 라이나손보 소식지 금액 확정 시 △ → O
