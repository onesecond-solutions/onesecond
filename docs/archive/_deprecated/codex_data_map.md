## 1. 큰 그림 — 무엇을 만드나

**자료함 정리기** = 사용자가 자료실에 자료를 올리면 코덱스가:
1. **자동 정리** — 분류·본문추출·태깅해서 **사용자 자료함에 정돈**(사용자에게 보이는 가치).
2. **데이터화** — 같은 자료를 검색/지식 자료로 활용(플랫폼 가치, 검색 인프라 연료).

→ 사용자 편의(딸깍 자동 정리) + 검색 인프라가 한 작업에서 동시 발생.

## 2. 🚨 사용자 자료 안전 원칙 (절대)

자료실 자료 = **남(사용자)의 자료**다. 사고 시 사용자 자료 손상 = 치명.

- **초기 = 원본 보존 + 제안/사본 방식.** 코덱스는 원본을 이동·삭제·덮어쓰기 하지 않는다. 정리 결과(분류·태그·추출 본문)는 **별도 컬럼/사본/제안 레코드**로 생성.
- 사용자 자료함에 **자동 반영(원본 재배치)**은 충분히 검증되고 **대표님 결재** 후에만.
- 의심되면 멈추고 보고. 사용자 데이터 쓰기는 가장 보수적으로.

## 3. 저장 지형 (코드 기준 — 코덱스가 실측 확인)

> 아래는 코드에서 파악한 골격. **코덱스는 작업 전 신버전에서 실제 스키마·RLS·경로를 SELECT로 확인**한다.

### 3-1. Storage 버킷
| 버킷 | 공개성 | 용도 |
|---|---|---|
| `myspace` | private | 사용자 자료함 파일 원본 (자료실 핵심) |
| `newsletters` | private | 보험사 소식지 원본(PDF/PPTX) |
| `onesecond_banner` | public | 배너·게시글 첨부 이미지 |
- 경로 패턴: `/storage/v1/object/{bucket}/{path}` (쓰기), `/storage/v1/object/sign/{bucket}/{path}`(서명 URL 읽기), public은 `/object/public/{bucket}/{path}`.
- 🚨 Storage 키는 **비ASCII(한글) 거부** → 키는 ASCII/uuid, 원본 한글명은 별도 컬럼(`source_filename`/`original_name`) 보존.

### 3-2. 자료 테이블
| 테이블 | 용도 | 비고 |
|---|---|---|
| `library` | 자료함 항목 메타 | owner_id·scope(private/public)·title·created_at·last_used_at |
| `myspace_files` | 자료함 파일 | 폴더 구조 |
| `myspace_folders` | 자료함 폴더 | 계층 |
| `newsletters` | 소식지 | 534건, OCR 본문화 대상(파일럿) |
| `knowledge_entries` | 검색 지식 | 적재는 **`status='ai_draft'`만**(검색 노출=승인 후) |
- RLS: 자료함 = `(auth.uid())::text = owner_id` 소유자 격리. 코덱스는 service_role로 접근하되 **소유권 경계를 코드로 존중**(남의 자료 함부로 X).

## 4. 수신 프로세스 (자료가 들어오는 길)

- 대부분 = **사용자들이 자료실 화면에서 직접 업로드** → `myspace` 버킷 + `library`/`myspace_files` 행 생성.
- 코덱스 트리거(검증 후): 새 자료 감지 → 정리 파이프라인 실행 → (제안/사본) 정리결과 + 데이터화.
- 구체 트리거 방식(폴링·webhook·배치 주기)은 파일럿 검증 후 총괄팀장과 확정.

## 5. 운영 리듬 — 하루 2회 결재 (2026-06-13 대표님)

- 코덱스는 작업 결과를 **PR로 쌓아둔다**(자동 머지 금지).
- 대표님이 **하루 2회(아침·저녁)** 결재 = 검수·승인·머지.
- 즉 코덱스는 **밤새 PR을 만들어 두고**, 대표님 결재 시점에 총괄팀장 검수 → 일괄 머지.
- 검색 노출(`approved` 승격)도 이 결재 흐름 안에서. 코덱스 단독 노출 금지.
