# 채굴 정규화 검수기준 v1 — 영구 진실원천

- 일자: 2026-06-20
- 발신: 총괄팀장(Code) / 결재: 임태성 대표님 (작업지시서 직접 하달)
- 성격: 채굴 파이프라인 전 구간의 **단일 진실원천**. 빌드 아님(문서 박제). DB·코드 변경 0.
- 선행: [[2026-06-19_search_mining_architecture_review]] Phase 1 선결조건 ① 충족 / 선결조건 ② B-2 정제본 품질 게이트는 2026-06-20 검수팀 검증 통과.

> 이 문서는 아래 전 구간의 영구 진실원천이다. 각 문서에 기준을 복사하지 말고 이 문서를 링크해 단일 소스로 유지한다.
>
> ```
> raw 원문 → OCR 정제본 → 정규화 지식 후보 → 자동 검증 → 사람 검수 → approved / hold / archive / discard
> ```

본 기준은 세 가지를 동시에 만족한다.
1. 현재 보험정보의 정확성·안전성 확보
2. 지난 소식지·단종·폐지 자료의 학습·역사 가치 보존
3. 자동채굴 검수 병목을 줄이면서, 원문에 없는 사실 생성 차단

---

## 1. 한 줄 원칙

> 고객·작성자·내부 조직의 식별정보는 제거하고, 보험회사·상품·담보·기준 등 **검증 가능한 보험 사실과 출처는 보존한다.**

주의:
- 작성자의 GA·지점·팀 소속은 제거 대상.
- 공식 출처인 **보험회사명은 제거 대상이 아니다.**
- 공식 상품명·담보명·특약명도 정확성 검증을 위해 보존.
- 특정 보험사를 무조건 지우는 식의 익명화 금지.

---

## 2. 소스 적격성 게이트 (정규화 검수 이전 단계)

정규화 검수 전에 별도 `source eligibility gate`를 둔다.

### 2-1. 공개 지식 채굴 허용 원천
- 원수사 공식 자료
- 심의번호·출처가 확인되는 심의필 자료
- 공식 약관·상품요약서·공시자료
- onesecond가 직접 작성하고 출처를 검증한 자료

### 2-2. 공개 지식 자동 채굴 금지 원천
- 타 GA가 자체 제작한 비심의 자료
- 타 설계사 개인 제작물
- 출처 불명 비교표·교육자료
- 작성·가공 권리 확인이 안 된 외부 저작물
- 지점 내부 운영자료 및 개인 의견

처리 방식:
```
공개 지식 자동채굴 금지
→ 지점 내부 검색·활용은 유지
→ 공개 편입 필요 시 별도 사람 검수
```

즉, **지점 자료실에서 검색되는 것**과 **`원세컨드 제공` 지식으로 공개되는 것**은 분리한다.

### 2-3. 소스 판정 결과
```
eligible_public         (공개 지식 채굴 가능)
internal_only           (지점 내부 검색만, 공개 금지)
manual_review_required  (사람 검수 후 결정)
source_rejected         (mine-batch 대상에서 제외)
```

`source_rejected` 원문은 mine-batch 대상에서 제외한다.

## 3. 개인정보 판정

개인정보가 발견됐다고 자료 전체를 무조건 폐기하지 않는다.

### 3-1. 즉시 공개 금지 정보
- 고객 실명 / 주민등록번호 / 전화번호 / 생년월일
- 계약번호·증권번호 / 주소
- 상담 녹취·카카오톡 대화 원문
- 가족관계·질병이력 등 특정 고객 사생활
- 작성자 실명·내부 연락망·작성자 소속 GA/지점/팀

### 3-2. 판정 분기
```
개인정보와 일반 보험지식을 안전하게 분리 가능
→ 개인정보 제거 → hold → 사람 검수 후 재판정

개인정보·개별 고객사례와 보험지식이 분리 불가능
→ discard
```

개인정보가 삭제됐다는 이유만으로 자동 approved 처리하지 않는다.

### 3-3. 보존해야 하는 공식 고유명사 (개인정보 아님 — 사실축 보존)
보험회사명 · 상품명 · 담보명 · 특약명 · 약관명 · 심의번호 · 공식 제도명

> 근거 사례: [[2026-06-19_search_mining_architecture_review]] §6 박영선님 카톡 — 고객 실명+상담 녹음+가족 얘기는 [3-1] 즉시 공개 금지 + 분리 불가능 = discard. [[customer_pii_consent_based_ok]]의 "동의 기반 저장 정상"은 본인이 수집·동의한 고객 기본정보 한정이며, 타인 상담 녹음·가족·타 고객 실명은 동의 밖.

## 4. 정확성 자동 diff 검증 (사람 표본검수 이전 단계)

기존의 사람 표본검수 **전에** 자동 diff 검증을 추가한다.

### 4-1. raw ↔ 정규화 자동 비교 대상 토큰
숫자 · 금액 · 비율 · 나이 · 기간 · 날짜 · 한도 · 면책·감액기간 · 보험회사명 · 상품명 · 담보명 · 특약명 · 제도명 · 심의번호

### 4-2. 자동 플래그 조건 (하나라도 발생 시 자동 approved 금지)
```
원문에 없던 숫자 또는 고유명사가 정규화본에 새로 생김
원문의 숫자·금액·기간이 변경됨
보험회사·상품·담보·특약명이 달라짐
조건이나 예외가 누락돼 의미가 달라짐
서로 다른 자료의 사실이 하나로 섞임
```

판정:
```
diff_pass     → 다음 검수 단계 진행
diff_warning  → hold(사람 확인)
diff_fail     → 자동 approved 금지, 재정제 또는 discard 검토
```

### 4-3. 구현 원칙 (과설계 금지)
완벽한 의미 비교를 처음부터 만들지 않는다. 1차 구현은 **보수적 플래그 방식**으로 시작한다.
- 숫자·날짜 정규식
- 금액·비율·기간 토큰
- 알려진 보험회사명 사전
- 상품·담보·특약 고유명사 후보

> 보수적 = 의심되면 통과시키지 말고 `diff_warning`으로 hold. 거짓 통과(false approved)보다 과검출(불필요 hold)이 안전.

## 5. 익명화·일반화 기준 (공개 지식 편입 조건 — 모두 통과)
- 고객·작성자·내부조직 식별정보 제거
- 특정 고객 1:1 설계안이 일반 원칙으로 재서술됨
- 원본 문장을 길게 복제하지 않고 사실축만 추출
- 공식 보험회사명·상품명 등 필요한 사실은 보존
- 원문 의미를 바꾸지 않음 (원문에 없는 일반론을 AI가 새로 만들어 넣지 않음)

예:
```
X: 더원지점 김○○ 설계사가 박○○ 고객에게 추천한 암보험 설계
O: 암 주요치료비 설계 시 진단비·수술비·치료비 담보를 함께 검토할 수 있다
```

---

## 6. 중립성 기준 (추천 금지 아님 — 공정 비교)

통과 조건:
- 특정 보험사를 소속·이해관계 때문에 우선하지 않음
- 비교 기준이 보험료·보장·인수조건·지급조건 등 사실에 기반
- 특정사 비방·근거 없는 배제 없음
- 광고성 문구를 객관적 지식처럼 변환하지 않음
- "무조건 최고", "반드시 가입" 등 단정적 영업표현 제거

```
허용: A사는 보험료가 낮지만 B사는 특정 담보 한도가 높다
금지: A사가 무조건 최고이므로 가입해야 한다
```

> 정합: [[strategy_neutrality_fair_comparison]] — 중립 = 공정 비교로 최선 찾기(추천 안 함이 아님).

## 7. 출처·근거 보존

모든 지식 후보는 **추적 가능한 근거 1개 이상** 필수.

필수 메타:
```
source_type
source_id 또는 source_path
source_company
published_at
review_number
valid_from
valid_until
last_verified_at
```

모든 필드가 처음부터 채워질 필요는 없으나 **원천 연결 자체는 필수**.

판정:
```
출처 연결 누락        → hold
출처 연결 완료        → 검수 계속
출처 허위·확인 불가   → discard
```

심의필 자료도 **원문 전체를 공용 지식으로 복제하지 않는다.** 원문은 근거 레이어에 두고, 공개 지식에는 사실축과 출처 연결만 둔다. (저작권 경계 — 아키텍처 회신서 선결조건 1 법률 확인과 정합.)

## 8. 시의성·아카이브 기준 (명백한 폐지·단종·만료를 자동 폐기하지 않음)

각 자료/지식 후보에 상태를 둔다.
```
current
archive
expired_or_superseded
validity_unknown
```

### 8-1. current
- 현재 유효성 확인됨
- AI의 현재 기준 답변에서 우선 근거로 사용 가능

### 8-2. archive
- 지난 소식지 / 과거 상품·인수기준·제도
- 학습·변천사·비교 가치 있음
- 현재 기준 답변의 **기본 근거에서는 제외**
- 사용자가 `지난 소식지 모음`, `과거 자료 포함` 등으로 접근 가능

### 8-3. expired_or_superseded
- 사용기한 만료 / 최신 자료로 대체됨 / 단종·폐지 확인
```
삭제 금지
현재 정보처럼 노출 금지
아카이브·과거 근거로 보존
최신 대체자료 있으면 연결
```

### 8-4. validity_unknown
- 유효기간·현재성 확인 불가
- approved 금지 → hold 상태에서 검수

> 원칙: **거짓·오염·가치 없음은 폐기하고, 과거의 사실은 아카이브한다.**

## 9. 최종 판정 4분기 (기존 3분기 → 4분기)

### ✅ approved
- 사실축 정확 / 자동 diff 통과
- 개인정보·내부 식별정보 제거 / 출처 연결
- 현재성 또는 사용 맥락 확인
- 검색 노출 가능

### 🟡 hold
- 개인정보 제거 후 사람 확인 필요
- 숫자·고유명사 diff warning
- 출처 연결 필요 / 현재·과거 판정 미확정
- 일부 문장 재정제 필요
- 자동 재채굴 또는 수동 수정 대상

### 📦 archive
- 과거·단종·폐지·만료 자료지만 학습·역사 가치 있음
- 현재 답변의 우선 근거에서는 제외
- 아카이브 검색·변천사 분석에는 활용
- 지난 소식지 모음에 노출 가능

### ❌ discard
- 보험 지식 가치 없음
- 개인정보와 지식을 분리할 수 없음
- 허위·왜곡
- 외부 저작물 소스 부적격
- 출처 허위 또는 확인 불가능
- 재사용 가치 없음

discard된 원천은 **재채굴 방지 마커**를 남긴다.

## 10. mine-batch 자동 가드 vs 사람 검수 역할 분담

mine-batch 자동 단계:
1. 소스 적격성 판정 (§2)
2. 개인정보·내부 식별정보 탐지 (§3)
3. 익명화·일반화 (§5)
4. 중립성 검사 (§6)
5. raw↔정규화 토큰 diff (§4)
6. 출처 메타 연결 (§7)
7. 현재성 후보 추출 (§8)
8. **approved 자동 확정 금지** — 기본 결과는 `ai_draft` 또는 `hold`

사람(대표님) 검수 역할:
- diff warning 확인
- 수치·상품·담보·예외 원문 대조
- 시의성 current/archive 판정
- approved/hold/archive/discard 최종 확정

## 11. 표본 검수 (대량 채굴 전 필수)

대량 채굴 전 반드시 소규모 표본을 먼저 검수한다.

표본 구성:
상품 자료 · 인수기준 · 심의필 자료 · 당월 소식지 · 지난 소식지 · 개인정보 포함 게시판 사례 · 타 GA 제작물 사례 · 만료·단종 자료

**최소 20건**을 raw↔정제본↔정규화 지식으로 대조.

통과 기준:
- 숫자 왜곡 0
- 신규 고유명사 생성 0
- 개인정보 노출 0
- 출처 연결 누락 0
- current/archive 오판 최소화
- 공개 부적격 소스 자동 편입 0

**표본 검수 통과 전 전체 mine-batch 가동 금지.**
