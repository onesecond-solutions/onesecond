## § 7. Code 5갈래 의견 (보류 시점 정리, 재논의 시 보강)

### 갈래 A: 데이터 구조 — 8 테이블 vs 통합

- AI 그림 5 테이블 (insurers / insurer_contracts / insurer_settings / insurer_documents / insurer_change_log) + CSV 3종(documents / document_pages / search_index) = **8 테이블**
- Code 의견: 별도 3 테이블 신설 권장 (library와 분리 — 용도가 다름)
- 핵심 결정점:
  - `insurers` 마스터 ↔ `users.company` 텍스트 컬럼 연결 방식 (FK? 자동 매칭?)
  - `insurer_documents`(메타) ↔ `documents`(CSV) 합치기 vs 분리?
  - `change_log` 1개 vs 각 테이블별 audit 컬럼?

### 갈래 B: 검색 시스템 — tsvector vs 외부 엔진

- (a) PostgreSQL `tsvector` + GIN 인덱스 — Supabase 내장, 추가 비용 0, 한국어 약함
- (b) Meilisearch / Typesense — 한국어 강함, $20~50/월 추가
- (c) Supabase Edge Function + OpenAI Embedding — 시맨틱 가능, 호출당 비용
- **Code 추천: (a) 일단 시작 → 약점 가시화 시 (b) 이전.** v1.5 진입 시점에는 (a)로 충분
- 핵심 결정점:
  - 검색 노출 등급(1순위/일반/후순위/차단) 구현 — `insurer_settings.search_rank` 정수값?
  - 카테고리별 1순위 vs 글로벌 1순위?

### 갈래 C: 어드민 컨트롤러 — D-9 vs 흡수

- (a) admin_v2 D-9 "원수사 운영" 신설 ⭐ — 사업 운영 영역 분리
- (b) D-1 users에 insurer 그룹 흡수 — 가독성 ↓
- (c) 별도 페이지 `pages/insurer-admin.html` — 통합성 깨짐
- **Code 추천: (a) D-9 신설.** admin_v2 사이드바 8섹션 → 9섹션 확장
- 핵심 결정점:
  - 5개 다이얼 UI — 행별 인라인 vs 별도 모달? (보류 결정 #4와 정합)
  - 대시보드 KPI 추가 (월 매출 / 유료 N社) — D-8 dashboard 흡수 vs D-9 자체?

### 갈래 D: 권한 분기 — 9역할 × 16조합 단순화

- 게시판 상태 4종 × 검색 노출 4종 = 16조합 × 9역할 = 144조합 → 명문화 불가능
- **Code 추천: 그룹화 단순화**
  - `admin` (전권)
  - `insurer_*` 4종 (자기 회사 자료만)
  - `ga_*` 4종 (모든 유료 자료 검색 가능, 게시판 작성 X)
- 핵심 결정점:
  - `insurer_*` 사용자가 **타사 자료** 보는지? (경쟁사 자료 비공개 vs 모두 공개)
  - `ga_staff` (스텝급)도 검색 가능? (현재 무료 혜택 범위 정합)

### 갈래 E: 자료 자산화 파이프라인 — 자동화 수준

| Phase | 작업 | 자동화 수준 |
|---|---|---|
| 1 (지금~5월) | 1년치 정리 | 팀장님 수동 (CSV 직접 임포트) |
| 2 (5~7월) | 신규 자료 업데이트 | 반자동 (PDF 업로드 → 페이지 분해 자동, 태깅 수동) |
| 3 (8월~) | 운영 단계 | 풀 자동 (Make.com + Claude API + Supabase 트리거) |

- 핵심 결정점:
  - PDF → page split + search_index 자동 생성 도입 시점 (Phase 2 vs Phase 3)
  - 정리 대행을 Make.com 워크플로우로 미리 구축 vs v2.0 구축?
