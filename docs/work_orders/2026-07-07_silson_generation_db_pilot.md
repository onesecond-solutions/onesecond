## 왜 실손 세대가 파일럿인가
- **독립 축** — 회사·상품 마스터에 안 붙는 "제도 지식"이라, 등뼈(상품 마스터) 미완 상태에서도 완결 가능
- **5 row 저위험** — 1~5세대. "원장 → DB → 화면 조회" 전 사이클을 작게 1회전 검증
- 스마트 설계서가 당장 실손 세대를 재사용(고객 실손 가입일 → 세대 판정 → 세대 정의 조회)

## 현재 하드코딩 위치 (전환 대상)
- **`pages/silson-generations.html`** — 1~5세대 정의 표(재가입주기·갱신·입원 자기부담·통원 공제/한도·입원 한도·비급여 구조·보험료 할증·한줄정리) + 세대별 카드/패널 전량 하드코딩(약 250줄). JS는 `sel(g)` 토글뿐, fetch 0.
- app.html:3523 — `#v-silson`이 위 파일을 **iframe으로 로드**만. app.html:6568 홈 카드 정의, 6944 VALID_VIEWS.
- app.html:6315-6317 — 실손 5세대 비교 카드 하드코딩 **잔재**(변천사 뷰와 별개, 중복 정리 대상).
- 참고: 페이블5 보장분석 HTML의 `GEN_DEF`(5세대 정의)도 같은 지식 — 스마트 설계서가 이 DB를 조회하게 될 축.

## 다음 세션 작업안에 포함할 7항목 (대표 지시)
1. **현재 하드코딩 위치** — 위 목록 확정(pages/silson-generations.html + app.html 잔재)
2. **JSON 원장 구조** — 세대별 필드 표준(gen·range·from/to·재가입·자기부담·통원공제·통원한도·입원한도·비급여구조·보험료할증·한줄정리 등). `judgeGen(date)` 판정 로직이 재사용할 from/to 경계 포함
3. **DB 스키마** — `silson_generations` 테이블(제도 지식·회사 무관 독립 축). approved/published 게이트 컬럼 포함
4. **approved/published 게이트** — 검수 전 화면 노출 금지. approved된 row만 화면·검색·스마트 설계서 재사용
5. **화면 조회 전환 방식** — `pages/silson-generations.html`을 DB 조회형으로(하드코딩 표 → DB fetch 렌더). 페이블5 세대 비교표(`genTableHTML`)도 같은 소스 재사용
6. **롤백 방법** — 신규 테이블·조회 경로라 기존 무영향. 게이트 플래그 off / 조회 실패 시 기존 정적 HTML 폴백 / 테이블 drop
7. **검수 항목** — 5 row 정확성(세대 경계일·자기부담·한도 수치), 화면 렌더 정합, 세대 판정(`judgeGen`) 경계 케이스

## 방향 확정 4건 (전체 로드맵 참조 — 순차)
1. 정적 파일럿 = **실손 세대**(본 작업)
2. 검수·발행 게이트 = approved/published 통일
3. 문자열→id 매핑 = 상품 마스터 확정 후 일괄
4. 동적 착수 = 원수사 자료실 검색 반영부터(cron/안전 큐·격리 필터 먼저)
