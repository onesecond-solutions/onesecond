# 작업지시서 — 비로그인 초기 2.5초 빈 화면 제거

> **이 지시서만 따르라. 범위 밖 작업 금지.**
> **담당:** 코상무(Codex) · **발주:** 총괄팀장 · **일자:** 2026-07-18 · **본가동 첫 작업**
> 행동 규칙 = 루트 `AGENTS.md`. 저장소·브랜치·배포 규칙 그대로.

## 1. 문제
비로그인 사용자가 사이트에 처음 들어오면 **초기 ~2.5초 동안 빈 화면**이 보인 뒤 홈이 렌더된다.

## 2. 원인 (확인됨)
`app.html`의 `startBoot()` (약 6218행):
```js
function startBoot(){
  if(getBootRole()){ bootView(); return; }              // role 있으면 즉시 렌더
  document.addEventListener('appstate:ready', bootView); // loadUser 완료 후 1회
  setTimeout(bootView, 2500);                            // 비로그인/실패 fallback
}
```
- **로그인 사용자**: `getBootRole()`이 role 반환 → 즉시 `bootView()` → 빈 화면 없음.
- **비로그인 사용자**: role 없음 → `appstate:ready`를 기다리는데, 비로그인은 `loadUser`가 role을 못 채워 이 이벤트가 안 뜬다 → **`setTimeout(bootView, 2500)` 만료까지(2.5초) 화면이 빈다.**

## 3. 목표
비로그인 초기 진입 시 **빈 화면 시간을 없애거나 최소화**(즉시/수백 ms 내 렌더). **로그인 사용자 진입은 회귀 0.**

## 4. 🚨 수정 범위 (절대 — 대표 확정)
- **`app.html`의 `startBoot()` 최소 변경만.** (필요 시 `bootView()` 중복 호출 방어 가드 정도까지는 허용하되, 그 외 함수·로직으로 번지지 말 것)
- **건드리지 말 것:** 게이트·권한(`_canSee*`·role 판정)·DB·Supabase·Secrets·`loadUser`·`bootView`의 렌더 로직 본체·다른 화면·CSS.
- **최소 변경 원칙:** 2.5초 폴백 상수를 줄이거나, 비로그인을 조기 감지해 빠르게 `bootView()`를 부르는 정도. 구조를 새로 짜지 말 것.

## 5. 구현 시 반드시 확인 (코상무 판단)
- **멱등성:** `bootView()`가 `appstate:ready`·`setTimeout`·즉시호출 등 **여러 경로로 중복 호출될 수 있다.** 중복 호출돼도 안전한지(이중 렌더·이벤트 중복 바인딩 없는지) 확인하고, 필요하면 `bootView` 1회 실행 가드를 넣어라(범위 안). **비로그인 폴백을 앞당길 때 로그인 경로와 폴백이 겹쳐 두 번 렌더되지 않게** 하는 게 핵심.
- **비로그인 판별:** 비로그인을 "빠르게" 알 수 있는 신호(토큰/세션 없음)가 있으면 그걸로 조기 렌더. 없으면 폴백 시간을 대폭 축소(예: 2500 → 짧게)하되, **loadUser가 실제로 늦게 끝나는 저속 네트워크에서 로그인 사용자가 폴백에 먼저 걸려 role 없이 렌더되는 회귀가 없는지** 반드시 따져라. (로그인인데 게스트 화면이 잠깐 뜨면 그게 더 나쁜 회귀다.)

## 6. 검증 (브라우저 실측 · 스크린샷/측정 근거)
- **비로그인 초기 진입**: 빈 화면 지속 시간 **전 vs 후** 측정(개선 확인).
- **로그인 사용자 진입**: 정상 홈 렌더, **role 화면(게이트) 회귀 0**, 게스트 화면 깜빡임 0.
- 저속 네트워크(throttle) 조건에서도 로그인 사용자가 role 없이 렌더되지 않는지.
- 콘솔 에러 0. `node --check` 불가(app.html은 HTML) → 브라우저 로드로 검증.
- 라이브 용어·기능 무변경(순수 타이밍 수정).
