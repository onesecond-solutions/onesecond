## 목표

마이 스페이스의 스크립트 글쓰기 화면에서 사용자가 자신의 다른 스크립트 카드를 검색·선택하여 본문 안에 내부 링크로 삽입하고, 링크 클릭 시 해당 스크립트 상세 화면으로 이동할 수 있게 한다.

## 이번 1차 범위

- 스크립트 → 스크립트 연결만 지원
- 본문 안에 내부 카드 링크 삽입
- 내 스크립트 검색·선택
- 링크 클릭 시 해당 스크립트 상세 화면 이동
- 브라우저 뒤로 가기 시 원래 스크립트로 복귀

## 이번 작업 제외

- 외부 URL 삽입
- 메모·자료실·공지·팀방·지점방 연결
- 글 하단 관련 카드 목록
- AI 자동 추천
- 새로운 통합검색 시스템
- DB 테이블 신규 생성
- 링크 미리보기 카드
- 다중 카드 일괄 선택

## 운영 원칙

- main 직접 push 금지
- 별도 브랜치 → PR → 정적 검수 → 머지
- 기존 스크립트 작성·수정·저장·조회 기능 회귀 금지
- 기존 RLS와 사용자 권한을 우회하지 말 것
- service_role 사용 금지
- 내부 링크 기능 때문에 스크립트 원문 구조를 대규모 변경하지 말 것

## 1단계 — 현재 구조 실측

코드 수정 전 아래를 확인하고 보고한다.

1. **실제 사용 중인 스크립트 글쓰기 화면**
   - 인라인 에디터 `scr-il-rich`
   - 모달형 또는 구형 에디터 `mysWriteScrText`
   - 현재 사용자에게 실제 노출되는 에디터가 무엇인지
   - 두 에디터를 모두 유지해야 하는지

2. **스크립트 데이터 구조**
   - 실제 테이블명
   - id 타입
   - 제목 컬럼명
   - 본문 컬럼명
   - 작성자 컬럼명
   - 공개·공유·개인 범위 컬럼
   - 삭제 상태 또는 비활성 상태 컬럼
   - 현재 상세 화면 진입 함수와 라우팅 방식

3. **저장 방식**
   - 본문이 HTML인지 텍스트인지
   - 저장 전 HTML 정화 함수
   - 검색용 평문 생성 방식
   - 수정 화면에서 기존 본문을 다시 불러오는 방식

4. **권한**
   - 사용자가 조회 가능한 스크립트를 결정하는 기존 RLS
   - 현재 사용자 자신의 스크립트만 검색할 수 있는 API 조건
   - 공유받은 스크립트까지 검색 대상에 포함되는지

→ 추정하지 말고 실제 코드와 DB 호출 근거로 확인한다.

## 2단계 — 1차 UX 구현

현재 실제 사용 중인 스크립트 글쓰기 툴바에 다음 버튼을 추가한다.

**버튼명:** `카드 연결`

**동작:**
1. 사용자가 본문에서 링크로 만들 텍스트를 선택
2. `카드 연결` 클릭
3. 작은 선택 창 또는 기존 UI 구조에 맞는 패널 표시
4. 사용자의 스크립트 제목 검색
5. 검색 결과에서 카드 1개 선택
6. 선택한 텍스트에 내부 링크 적용

**텍스트를 선택하지 않은 경우:**
- 선택한 스크립트 제목을 본문 커서 위치에 링크 텍스트로 삽입

**검색 결과 표시:**
- 스크립트 제목
- 유형(콜 스크립트/채팅 스크립트)
- 수정일 또는 생성일
- 동일 제목 구분에 필요한 최소 정보

**검색 대상:**
- 우선 사용자가 열람 가능한 스크립트
- 최소 기준은 본인 소유 스크립트
- 기존 RLS를 그대로 통과한 결과만 표시
- 삭제·비활성 스크립트 제외
- 현재 작성 중인 자기 카드 연결 제외

## 3단계 — 내부 링크 저장 규칙

일반 웹 URL을 내부 링크처럼 저장하지 말고, 원세컨드 내부 링크임을 식별할 수 있어야 한다.

**권장 형태:**
```html
<a href="#" data-os-link-type="script" data-os-link-id="실제ID">표시문구</a>
```

**필수 조건:**
- `data-os-link-type="script"`
- `data-os-link-id`에 실제 DB id
- 링크 제목이 바뀌어도 id 기준 연결 유지
- id 타입을 임의로 uuid 또는 bigint로 단정하지 말고 실측값 사용
- `javascript:` URL 저장 금지
- 사용자 입력값은 기존 정화·이스케이프 경로 통과
- 검색용 평문에는 링크 표시문구만 자연스럽게 남도록 처리

`href`에는 외부 URL을 넣지 않는다. 현재 SPA 라우팅 방식과 충돌하지 않는 안전한 값 또는 내부 경로를 사용한다.

---

## 4단계 — 링크 클릭 동작

스크립트 상세 화면에서 내부 링크를 클릭하면:
1. 기본 브라우저 이동 차단
2. `data-os-link-type`과 `data-os-link-id` 확인
3. 기존 스크립트 조회 API로 대상 재조회
4. 현재 사용자에게 조회 권한이 있을 때만 상세 화면 이동
5. 새 모달을 만들지 말고 현재 프로젝트의 상세 페이지 이동 방식을 재사용
6. 브라우저 뒤로 가기 시 원래 스크립트와 스크롤 위치 복귀

**링크 대상이 없는 경우:**
- "삭제되었거나 접근할 수 없는 스크립트입니다." 안내
- 빈 화면이나 무한 로딩 금지

**권한이 없는 경우:**
- 제목이나 본문을 노출하지 말고 동일한 접근 불가 안내 사용
- ID만으로 자료를 직접 가져오지 말고 RLS 적용 조회 사용

---

## 5단계 — 수정 화면 보존

내부 링크가 포함된 기존 스크립트를 다시 수정할 때:
- 링크가 사라지지 않아야 함
- `data-os-link-type`, `data-os-link-id`가 유지돼야 함
- 링크 표시문구 수정 가능
- 일반 본문 수정과 저장 정상
- AI 글 다듬기가 링크 태그나 대상 id를 파괴하지 않아야 함

**AI 글 다듬기와 충돌 가능성이 있으면:**
- 내부 링크를 임시 토큰으로 보호한 뒤 결과에 복원하거나
- 링크 영역을 AI 변환 대상에서 제외

→ 링크가 깨질 가능성이 있으면 임의 처리하지 말고 원인과 최소 해결안을 보고한다.

## 총괄팀장 사전 메모 (착수 전 확인 포인트)

1. **에디터 이원화 확정이 1단계 핵심** — 인라인 `osRich`(`scr-il-rich`)와 모달형(`mysWriteScrText`) 둘 다 존재. 실사용 노출 에디터 1개에 먼저 버튼 부착(둘 다면 작업량 2배 → 결재 포인트).
2. **id 타입 단정 금지** — 6/22 공유 버그 때 `scripts.id`=bigint 신호(myspace_files=uuid). 실측 확정 후 `data-os-link-id`는 실측 타입 그대로 저장.
3. **AI 글 다듬기 충돌** — `_aipMenu`가 본문을 Gemini 왕복시켜 링크 `<a data-os-link-id>` 파괴 위험. ai-polish 입출력 경로 실측 후 토큰보호/변환제외 중 최소안 보고.
4. **`href="#"` + SPA 라우팅 충돌** — 해시 라우터(`showView`/pushState)와 `#` 충돌 주의. 클릭 핸들러 `preventDefault` 확실히.
5. **평문 인덱스** — `search_text` 생성 시 링크 표시문구만, 태그/속성 제거(스크립트 HTML 정화 트랙 정합).
