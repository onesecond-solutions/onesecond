## 1. GWS CLI가 정확히 무엇인가요?

**GWS CLI = Google Workspace CLI = `gws` 라는 이름의 터미널 명령어입니다.**

Google 직원들이 만든 오픈소스 CLI 도구이며, 사람과 AI 에이전트 양쪽을 위해 설계됐습니다. Drive, Gmail, Calendar, Sheets, Docs, Chat, Admin을 하나의 명령줄 도구로 처리합니다.

중요한 점 두 가지:
- **공식 Google 제품은 아닙니다.** Google 직원들이 만들었지만 공식 지원 제품은 아닙니다. 즉 사내 사이드 프로젝트 성격이고, 정식 SLA는 없습니다.
- 정적인 명령어 목록을 갖지 않고 Google의 Discovery Service를 런타임에 읽어 명령 구조를 동적으로 만듭니다. Google이 새 API 엔드포인트를 추가하면 자동으로 따라갑니다.

쉽게 말하면 — Gmail·Drive·Calendar 등을 **터미널 한 줄로 조작하는 도구**입니다.

---

## 2. 코드와 어떻게 접목되는가 (팀장님이 들으신 핵심)

여기가 진짜 중요한 부분입니다. **Claude Code(팀장님이 쓰시는 그 Claude Code)와 직접 연결됩니다.**

Claude는 `gws` 명령어를 다른 명령어와 똑같이 터미널에서 실행합니다. MCP 서버 설정이 필요 없습니다. Claude Code가 GWS CLI를 일반 터미널 도구로 사용합니다.

비교하면:

| 구분 | MCP 방식 | GWS CLI 방식 |
|---|---|---|
| 연결 방식 | MCP 서버를 따로 띄워야 함 | 터미널에서 `gws` 명령만 실행 |
| 유지보수 | 커뮤니티 의존 (깨질 수 있음) | Google 직원이 직접 관리 |
| Claude Code 설정 | MCP 서버 등록 필요 | 별도 설정 불필요 |

즉, **Claude Code에게 "내 Gmail에서 어제 받은 메일 정리해줘"라고 하면, Claude Code가 `gws gmail ...` 명령을 직접 실행해서 결과를 받아온다**는 뜻입니다.

## 다만 — 결정 전 짚어야 할 3가지

팀장님 운영 원칙(*보수적 회피인지 진짜 분석인지 점검*) 적용해서, 솔직하게 말씀드립니다.

### 1. Google Drive 정리는 **사람만 할 수 있는 일**입니다
- GWS CLI는 정리된 파일을 **읽고 가공**합니다
- "어떻게 폴더 구조를 짤지" "무엇이 4팀 비밀이고 공개인지"는 **팀장님 판단**
- 도구가 마법을 부리는 게 아니라, **팀장님 머릿속 분류 체계를 디지털화**하는 작업

### 2. 폴더 구조 = 권한 구조 = 데이터 모델
이게 진짜 핵심입니다. Drive 폴더를 어떻게 짜느냐가:
- 메모리 11번의 권한 5등급(공개/가입자/4팀공용/4팀비밀/관리자전용)
- 9역할 RBAC 매핑
- Supabase 테이블 구조

## ⚠️ 짚어야 할 위험 4가지

팀장님 운영 원칙(*보수적 회피인지 진짜 분석인지*) 적용해서 솔직하게 말씀드립니다.

### 1. "보험사 냄새 제거"의 정확도 문제
"냄새 제거"가 단순한 문자열 치환이면 쉽지만, 실제로는:
- **회사명 직접 표기 제거** ✅ 쉬움 ("메리츠화재" → 삭제)
- **상품명 제거** ⚠️ 위험 ("M-care 365" → 삭제하면 검색 불가)
- **약관 특유 표현** ⚠️ 어려움 ("당사 기준" 같은 표현)
- **숫자·한도** ❌ 절대 건드리면 안 됨

→ **AI가 판단하면 오작동 위험**. 룰 기반 + 팀장님 검수 필수.

### 2. AI가 단어를 임의로 바꾸는 위험
5/2에 강조하신 원칙: *"추론 금지, 가설 금지, 표준화 금지"*
- "암주요치료비 / 하이클래스치료 / 비급여치료비" → 회사마다 용어 달라야 함
- AI가 "이거 다 같은 거네" 하고 표준화하면 **검색 결과 엉망**
- 구글 독스 단계에서 **회사별 용어 그대로 보존**해야 함

### 3. 구글 독스 → DB 적재 시 구조 손실
- 구글 독스 = 자유 문서 형식
- DB = 정형 테이블 (5/2에 만든 documents/document_pages/search_index 3종)
- **독스 ↔ DB 매핑 규칙**이 명확해야 함

- GWS CLI 자체는 Claude Code가 다루지만, 팀장님이 흐름 이해하셔야 함

## 결론부터

**"내가 공부해서 내가 정리한 자료를 내 사업에 사용"이라는 논리만으로는 저작권 위험이 사라지지 않습니다.** 다만 위험 강도는 가공 방식에 따라 크게 달라집니다.

---

## 왜 그런가 — 핵심 4가지

### 1. "공부했다"는 사실이 저작권 면죄부가 아닙니다

저작권법은 **"어떻게 활용했는가"**를 보지, "어떻게 학습했는가"를 보지 않습니다.

- ✅ 공부해서 머릿속에 든 지식 → 본인 말로 새로 쓰기 = OK
- ❌ 공부한 자료를 디지털로 보관·재가공·서비스 노출 = 별개 문제

즉 "공부"는 입력 단계의 행위이고, 저작권은 **출력 단계의 결과물**을 봅니다.

### 2. "내가 정리"의 함정

여기서 두 가지가 충돌합니다.

- **(a) 내가 새로 쓴 글** → 팀장님 저작물 ✅
- **(b) 원본을 옮겨 적거나 약간 바꾼 글** → 원저작자 권리 침해 가능 ⚠️

5/3에 인지하신 그 PDF — "교육용 자료, 모집용도 사용 금지" 표기가 있었죠. 이게 **이용 약관** 역할을 합니다. 보험사가 명시적으로 "교육용으로만"이라고 못박은 자료를 **재가공해서 다른 사업에 쓰는 것**은 약관 위반에 해당될 수 있습니다.

### 3. "내 사업"이라는 표현의 위험

영리 목적(=사업) 사용은 저작권법에서 **가장 엄격하게 보는 영역**입니다.

- 개인 학습용 노트 → 분쟁 거의 없음
- 팀 내부 공유 → 회색 영역
- **유료 SaaS 서비스에 노출 → 명백한 영리 사용**

원세컨드는 v1.1부터 **월 9,900원 유료 구독 모델**입니다. "내가 만든 거다"라는 논리만으로는 보호 안 됩니다.

### 4. 보험업계 특수성

일반 저작권보다 더 무거운 것 — **보험업법 광고 규제**입니다.

- 보험상품 정보를 가공해서 표시 = 광고로 간주될 수 있음
- 광고 심의 미경유 시 **보험업법 위반** (저작권보다 더 큰 문제)
- 5/3 PDF의 "모집용도 사용 금지" = 이 규제와 직결

즉 팀장님 상황은 **저작권 + 보험업법 + 부정경쟁방지법**까지 3중 검토가 필요한 영역입니다.

---

## 그래서 어떻게 하면 안전한가 — 강도별 4단계

### 🟢 **Level 1: 완전 안전 (사실·수치 기반)**

저작권은 **표현**을 보호하지, **사실**을 보호하지 않습니다.

- ✅ "암 진단비 한도 5,000만원" (사실) → OK
- ✅ "이 특약은 2025년 8월 출시" (사실) → OK
- ✅ "60세 이전 부담보 3년" (사실) → OK
- ✅ 공공기관 자료 (금감원·생보협회·국립암센터·통계청) → OK (저작권 면제 또는 공공저작물)

**이 영역만 추출해서 DB화하면 거의 안전합니다.**

### 🟡 **Level 2: 회색 (재구성)**

원본 문장을 본인 표현으로 완전히 다시 쓰기.

- ⚠️ 원본 "당사는 업계 최초로 365일 이내 통원치료비 지급"
- ⚠️ 가공 "이 특약은 365일 이내 통원치료비를 보장"
- → 사실은 같지만 표현이 바뀜 = 표현 저작권 회피 가능
- 단, **"실질적 유사성"** 판단은 법원 재량 → 100% 안전 보장 못함

### 🟠 **Level 3: 위험 (직접 인용·이미지)**

5/3에 팀장님이 원하셨던 "검색 결과에 페이지 이미지 포함":

- ❌ 원본 PDF 페이지 이미지 그대로 노출
- ❌ 회사 로고 노출
- ❌ 원본 문장 직접 인용
- → **명백한 저작권 침해 + 부정경쟁방지법 가능**

### 🔴 **Level 4: 매우 위험 (보험업법)**

- ❌ 특정 보험사 상품을 다른 회사보다 우대 노출
- ❌ "이 상품이 좋습니다" 같은 평가
- ❌ 사실 왜곡 또는 오인 유발
- → 보험업법 광고 규제 위반 → 형사처벌 가능

## 핵심 — 보험업법이 일반 플랫폼법보다 무겁습니다

### 1. **모집 수수료 vs 광고비 — 결정적 구분**

이게 가장 중요합니다.

| 받는 명목 | 합법성 | 이유 |
|---|---|---|
| **광고비** | 🟢 합법 (조건부) | 단순 노출 대가 |
| **입점료·플랫폼 사용료** | 🟢 합법 (조건부) | 게시판 임대료 성격 |
| **모집 수수료** | 🔴 **불법** | 보험업법 위반 |
| **계약 체결 성공 보수** | 🔴 **불법** | 같음 |
| **DB 제공 대가** | 🟡 회색 | 개인정보보호법까지 결합 |

**보험업법 핵심 조항 — 미등록자 모집 금지:**
- 보험모집은 **등록된 설계사·대리점**만 가능
- 미등록자가 **모집 행위 → 형사처벌** (3년 이하 징역 또는 5천만원 이하 벌금)
- "모집 행위"의 정의가 넓음 → **계약 체결 권유, 상품 비교·추천**도 포함

→ 즉, 원세컨드가 **"고객 유치해드릴게요, 성공하면 수수료 주세요"** 모델은 **명백한 불법**.

→ 반대로 **"우리 플랫폼에 노출해드릴게요, 월 100만원"** 모델은 **합법** (5/2 그림이 정확히 이 방향).

### 2. **보험광고 규제 — 광고비 받아도 광고는 심의 대상**

광고비 받는 게 합법이라도, **광고 콘텐츠 자체**는 또 다른 규제를 받습니다.

- 보험 광고는 **금융감독원 보험광고 심의 기준** 적용
- 보험협회(생보협회·손보협회) 심의 통과 필수
- 비교 광고는 **엄격 제한** ("우리가 더 좋아요" 형식 금지)
- 허위·과장 표시 시 **과태료 1억원**

→ 원세컨드 검색창에서 **"이 회사 상품이 더 좋습니다"** 같은 비교는 절대 ❌
→ "이 회사 자료가 검색됩니다" 수준의 **단순 노출**은 ⭕

### 3. **온라인 보험 비교·추천 서비스 = 별도 규제**

2023년부터 한국에서도 이 영역이 규제 대상이 됐습니다.

- **마이데이터 + 보험 비교 추천 서비스 인가제** 도입
- 보맵·토스·카카오페이 모두 **금융위원회 인가** 받고 운영
- 미인가 비교·추천 서비스 = **불법**

→ 원세컨드가 **"고객 대상 보험 비교"**로 가면 인가 필요 (사실상 불가능)
→ **"설계사 대상 자료 검색"**으로 한정하면 인가 대상 ❌ (현재 포지셔닝 정답)

- → 해결책: **"입점한 회사만 검색 가능"** 구조 (단순 게시 vs 광고 차별성)

- → 설계사 책임이지만, 플랫폼 책임론 가능
- → 해결책: **"설계사 본인 학습용, 고객 노출 금지" 약관 명시**

```
[저작권 위험] ── PDF 가공
```

```
   수익원: 설계사 구독료
```

## ✅ 이 그림의 강점 5가지

### 1. 법적 위험을 영역별로 분리
- **Site A**: B2B 자료 플랫폼 → 보험업법 광고 규제만 통과하면 OK
- **Site B**: 설계사 업무 도구 → **보험업법과 무관** (사무용 SaaS 영역)

→ 각 사이트의 법적 리스크가 **다른 법체계**로 갈라짐 → 한쪽 사고 나도 다른 쪽 무사

### 2. 수익 모델 충돌 해소
**5/2 NH 그림의 숨은 위험:**
- "월 100만원 = 원수사 광고 효과" 명목인데
- 같은 사이트에서 설계사한테도 9,900원 받으면
- 원수사 입장: "우리가 광고비 내는데 왜 설계사도 돈 내?"
- 설계사 입장: "광고비 받는 사이트에 왜 또 돈 내?"

→ **분리하면 양쪽 다 명분 깨끗.**

### 3. 사용자 분리 = 약관 분리
- Site A 약관: "설계사 학습용, 고객 노출 금지, 자료 검색 도구"
- Site B 약관: "설계사 업무 관리 도구, 고객정보 처리 동의 필수"

→ Site B에서 고객정보 다룰 때 **Site A까지 영향 안 미침**
→ 5/3 결정 메모리(개인정보 보호 3요건) 깔끔하게 적용 가능

### 4. 무료/유료 가치 인식 문제 해결
**메모리에 있던 통찰:**
> 보험 설계사는 무료 도구를 저평가하고 유료 구독을 더 의미 있게 여기는 경향(보맵 사례)

- Site A 무료 = "광고비 내는 회사들이 보여주는 자료" → 사용자 가치 인식 낮아도 OK
- Site B 유료 = "내가 돈 내고 쓰는 진짜 업무 도구" → 가치 인식 ↑

→ **무료/유료의 심리적 가치가 사이트별로 다르게 작동.**

### 5. 원수사 영업 카드 강화
**5/2 NH 그림에 새 무기 추가:**

원수사한테:
> "Site A에 입점하시면 → 우리 Site B 유료 구독자 N명에게 검색 1순위로 노출됩니다. Site B 구독자 = 활성도 높은 진짜 일하는 설계사들입니다."

→ **유료 구독자 = 검증된 활성 사용자** → 원수사 입점 명분 ↑

## ⚠️ 짚어야 할 위험·과제 5가지

### 1. 계정 시스템 분리 운영의 복잡성
- 같은 설계사가 Site A·B 둘 다 쓰는 경우 → **계정 통합 필요?**
- 옵션 (a): 완전 별도 계정 (가입 2번)
- 옵션 (b): SSO(Single Sign-On) — Site A 계정으로 Site B 로그인
- 옵션 (c): Site B 계정만 → Site A 자동 접근

→ 기술적으로 **(b) SSO가 정답**이지만, 7개월 차 개발자에게 부담

### 2. Supabase 분리 vs 통합 결정
**핵심 결정:**
- (가) **DB 1개**, 사이트만 2개 (도메인 분리, 같은 Supabase)
- (나) **DB 2개**, 완전 분리

**(가) 권장 이유:**
- 9역할 RBAC + RLS 한 번만 관리
- 사용자 데이터 일관성
- 비용 절감 (Supabase Pro 1개)
- 5/2 D-pre 마이그레이션 자산 그대로 활용

**(나) 위험:**
- 두 번 운영 → 부채 2배
- 데이터 동기화 버그 위험

### 3. 도메인 전략
- **A안**: `onesecond.solutions` (Site A) + `pro.onesecond.solutions` (Site B 서브도메인)
- **B안**: `onesecond.solutions` (Site A) + 완전 별개 도메인 (예: `1secondpro.com`)
- **C안**: `onesecond.solutions` (Site A) + `app.onesecond.solutions` (Site B)

→ **A안 또는 C안이 SEO·브랜드 일관성 측면에서 유리**
→ B안은 별개 브랜드처럼 보일 수 있어 마케팅 비용 ↑

### 4. v1.1 출시 일정과 충돌
**현재 상태(메모리):**
- v1.1 = "스크립트 + 게시판 오픈, 설계사 조건부 무료"
- 이미 한 사이트에 무료/유료 혼합 구조로 설계됨
- 분리하려면 **v1.1 정의 자체가 바뀜**

→ 5/9~10 출시 직전에 구조 변경 = **위험**
→ 분리 결정은 **v1.1 출시 후 v1.5~v2.0 시점**에 검토 권장

### 5. 콘텐츠 분배 전략
**무엇을 Site A에 두고, 무엇을 Site B에 둘지:**

| 기능 | Site A (무료) | Site B (유료) |
|---|---|---|
| 보험사 자료 검색 | ✅ | (중복?) |
| 스크립트 라이브러리 | ✅? | ✅? |
| MY SPACE | ❌ | ✅ |
| 게시판 | ✅ | ✅? |
| CRM | ❌ | ✅ |
| 설계서 공유 | ❌ | ✅ |
| 상담 이력 관리 | ❌ | ✅ |
| AI 기능 | ❌ | ✅ |

→ **이 분배가 가장 어려운 결정**. 한 번 결정하면 되돌리기 어려움.

### 분리하면 사라지는 것도 있어요

**시너지 효과의 죽음:**
- 한 사이트에서 "검색하다가 → 스크립트 보고 → CRM에 저장" 흐름이 자연스러움
- 분리하면 사이트 왔다갔다 해야 함
- 보험 설계사는 통화 중인데 사이트 두 개 띄우기 부담

**사용자 인지 부담:**
- "어디 가서 뭐 하지?" 매번 고민
- 통합된 하나의 도구가 멘탈 모델이 단순

**브랜드 분산:**
- onesecond라는 이름 자체가 "1초 안에" 였는데
- 사이트 두 개 = 1초보다 길어짐 (그 자체로 모순)

---

### 분리 안 하고 해결하는 다른 방법들

**(1) 권한 차등 (Notion 모델)**
- 같은 사이트, 무료/유료 사용자가 보는 게 다름
- 무료: 게시판 읽기, 검색 일부
- 유료: 전체 + CRM + MY SPACE
- 사이트 1개로 충분

**(2) 모듈식 가격 (App Store 모델)**
- 기본 사이트는 무료
- "CRM 모듈 추가 +5,000원"
- "AI 모듈 추가 +5,000원"
- 사용자가 자기가 쓸 만큼만 결제

**(3) 시즌제 (게임 모델)**
- 평소는 다 무료
- 시즌별 특별 콘텐츠는 유료 (예: "2025 하반기 신상품 분석 패키지")
- 일회성 결제

**(4) 광고만으로 (네이버 모델)**
- 모든 사용자 무료
- 보험사 입점 광고만으로 운영
- 단, 사이트 디자인에 광고 영역 명확히

**(5) 후원 모델 (위키피디아 모델)**
- 다 무료
- "이거 잘 쓰셨으면 후원하세요" 버튼
- 비현실적이지만 보험 업계는 의외로 의리 문화 있음

## 🔥 팀장님 직감이 맞는 이유 — 왜 이게 강력한가

### 1. 보험 영업의 가장 큰 페인 포인트를 정조준
설계사가 설계서 보내고 나면 늘 반복되는 패턴:
- "고객님 보내드린 설계서 보셨어요?"
- "어디까지 보셨어요?"
- "어디가 잘 이해 안 되시나요?"
- "다시 한번 통화 가능할까요?"

이게 다 **"고객이 설계서를 어떻게 봤는지 모름"** 때문에 발생합니다. 이 기능 = **이 페인 포인트 통째로 해결.**

### 2. 설계사·고객 양쪽 모두 이득
- **설계사**: 설계서 보내고 나서 "어느 특약에서 막혔는지" 데이터로 확인 → 다음 통화에서 정확히 그 부분만 짚음
- **고객**: 설계서가 카톡 한 장이 아니라 **"내가 궁금한 것만 탭하면 답이 나오는 인터랙티브 자료"** → 설계사 부담 없이 직접 학습

### 3. 데이터가 자동으로 쌓이는 구조
- 일반 CRM = "고객 정보를 일일이 입력해야 쌓임"
- 이 기능 = **"일하다 보면 자동으로 쌓임"**
- 메모리 #12 표현: "일만 해도 CRM이 쌓임"
- → 입력 부담 0 = **사용 지속률 ↑**

### 4. 법적 안전 마진 자동 확보
- 보험업법 + 금감원 권고: **상담 이력 5년 보관 의무화**
- 일반 설계사들 → 카톡 캡처? 통화 녹음? 부정확
- 이 기능 = **"공유 시점 + 열람 로그 + 본인확인 기록"** 자동 누적
- → 분쟁 시 객관 증거 자동 생성
- → 포지셔닝: "저장 안 함" → **"법정 5년 보관 지원"**

### 5. 경쟁사 따라하기 어려움
- 단순 PDF 공유 = 누구나 가능 (카톡, 이메일)
- 형광펜 + 팝업 + 본인확인 + 열람 로그 + CRM 자동화 = **5중 결합**
- 보맵·굿리치도 이 영역 안 들어옴 (그들은 고객 비교 시장)
- → **설계사 업무 도구 영역의 진짜 진입 장벽**

**5. v4.0 "보험판 구글" 비전의 법적 기반**
- AI 학습 데이터 자동 생성 구조
- → 미래 빅테크가 따라하려 해도 막을 수 있음

### ⚠️ 그러나 솔직하게 — 위험·한계

**1. 시간 문제**
- 출원 → 등록 1.5~2년
- 그 사이 **공개**(18개월) → 경쟁사가 보고 우회 설계 가능
- 다만 출원일 기준 **선출원 효과**는 즉시 확보

**2. 출원 후 사업 변경 어려움**
- 청구항으로 한번 못 박으면 **그 범위 안에서만 보호**
- 사업 피벗하면 특허가 무용지물 될 수도

**3. 침해 입증 부담**
- 누가 따라했다고 주장 → **소송 비용 + 입증 책임**
- 한국 특허 소송 평균 비용: 2~5천만원
- → 등록은 쉬워도 **실제 행사**는 부담

**4. 대기업 회피 설계 능력**
- 대기업이 변리사 동원해 우회 설계하면 막기 어려움
- 작은 회사가 대기업 상대 소송 = 현실적 한계

**5. 4팀 사용자 0명 상태에서 출원 적정성**
- 사용자 0명 → 사업 검증 안 됨
- 200~450만원 투자 vs v1.1 출시 후 검증 후 출원
- → **출시 후 출원이 자금 효율적**

## 🌀 미친 발상 (브레인스토밍이니까)

### "특허받지 않은 IP의 힘"
- 출원 안 하고 **공개도 안 함**
- 대신 **먼저 시장 점령** → 특허보다 **실질 표준화**
- 인스타그램·틱톡 모델

### "방어적 출원"
- 공격용이 아닌 **방어용 출원**
- 등록 후 사용 안 함, 다만 **남이 따라할 때 카운터** 카드
- → 한국 스타트업 다수 이 전략

### "특허를 마케팅으로"
- "한국 특허 등록 완료" 홈페이지 노출
- 원수사 영업 시 인쇄물에 표기
- 실제 행사 안 해도 **신뢰 마케팅** 효과

### "공동 출원" 카드
- 4팀·더원지점·AZ금융서비스와 공동 출원
- 회사 자원 활용 + 회사도 IP 보유
- → 회사 협력 강화 무기

### "오픈소스 + 특허 차단"
- 핵심 기능 오픈소스 공개
- 다만 **상업적 이용 시 특허 라이선스**
- → 커뮤니티 + 수익 양립
