- **#766 검색 현황 칩 어드민 전용 + 캘린더 검색 제외** — '검색 가능한 데이터' 칩(시스템 인벤토리)을 `#app.is-admin` 게이트로 일반 사용자 숨김(카운트 쿼리도 실행 전 차단). 캘린더(calendar_events)를 검색 결과 그룹·칩 양쪽에서 제거. 스크립트 칩 카운트를 실제 검색되는 `scope=global`만으로 정합.

## 🚨 검색 누출 점검 — 결론: 데이터 교차 누출 0건 (전 소스 RLS 실측 정상)

대표님이 일반 팀장 계정 교차검증에서 "검색에 다른 사용자가 봐선 안 될 내용 노출"을 제기 → 전수 점검.

**pg_policies 라이브 실측 (검수팀 실행) — 3개 소스 모두 정상 격리:**
- **calendar_events**: 개인 일정 `source_type='personal' AND author_id=auth.uid()` = 본인만 / 공개·팀·지점 audience_target별 소속 격리.
- **myspace_files** (`myspace_files_scoped` ALL): personal=`owner_id` 본인 / team=`scope_id=os_user_team_id()` / branch=`scope_id=os_user_branch_id()` / 어드민 팀·지점 읽기(admin_read).
- **team_notices** (`team_notices_select`): team_internal=`my_team_id()` / branch_internal=`my_branch_id()` / 어드민.

→ **일반 사용자는 애초에 남의 개인/타팀/타지점 데이터를 SELECT 못 함. 데이터 누출 없음.**

**대표 우려 "누출"의 실제 정체 = 데이터가 아니라 노출 표면:**
1. 어드민 인벤토리 칩이 일반 사용자에 노출(#766 차단) ← 진짜 문제
2. 개인 일정·자료가 검색 표면에 떠 있던 것(#766·#767 정합)
3. 옛 라벨 잔존(#768)

→ 캘린더 때와 동일 패턴: **데이터는 RLS로 안전, 노출 표면이 문제**. #767 자료실 방어 필터도 RLS가 이미 막아 실제 누출은 없었으나 무해한 한 겹 더(남겨둠).

## OCR / 보류함 운영 현황

- **검수팀 ocr-batch 재배포 완료** — raw main 코드(#762 서명URL 양방향 + #764 크기초과 마킹 포함) 배포 확인. net._http_response 200. 라이브 반영됨.
- **현재 보류 5건 전부 크기초과(18MB+)**: 소식지 3(메트라이프·KB손보 등, title=null이나 company fallback로 표시) + 자료실 2(미라클러닝 PDF). **대표 결정 = 그대로 유지, 나중 처리.**
- 크기초과 해소는 마킹 리셋으로 안 됨(18MB=Gemini inlineData 구조적 한계). 진짜 해소 = (A)18MB↓ 축소 재업로드 (B)대용량 OCR 경로 신설(Gemini File API/PDF 분할)=미래 트랙.
