## 9. Supabase 설계 (1차 가설)

### 9-1. 테이블 (기존 board 확장 또는 신규)

```sql
-- 옵션 A: 기존 boards 테이블 확장
ALTER TABLE boards ADD COLUMN scope TEXT DEFAULT 'public';
-- scope = 'public' | 'team4_public' | 'team4_secret' | 'admin'

-- 옵션 B: 신규 team4_vault_categories 테이블
CREATE TABLE team4_vault_categories (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  description TEXT,
  icon TEXT,
  display_order INT,
  is_active BOOLEAN DEFAULT true,
  created_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE team4_vault_posts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  category_id UUID REFERENCES team4_vault_categories(id),
  title TEXT NOT NULL,
  content TEXT,
  attachments JSONB,
  pinned BOOLEAN DEFAULT false,
  created_by UUID REFERENCES users(id),
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);
```

→ **결정 T-6과 연동하여 최종 결정**

### 9-2. RLS 정책 (예시)

```sql
-- 4팀 사용자만 조회 가능
CREATE POLICY "team4_only_select" ON team4_vault_posts
  FOR SELECT
  USING (
    auth.uid() IN (
      SELECT id FROM users WHERE team = 'AZ-더원-4팀'
    )
  );

-- 매니저 이상만 작성 가능 (선택)
CREATE POLICY "team4_manager_insert" ON team4_vault_posts
  FOR INSERT
  USING (
    auth.uid() IN (
      SELECT id FROM users 
      WHERE team = 'AZ-더원-4팀' 
        AND role IN ('ga_manager', 'ga_branch_manager', 'admin')
    )
  );
```

- **완화책:** "곧 일부 자료는 공개됩니다" 안내 + 점진적 공개 시각화

### ⚠️ 위험 2: 카운터·진척도 가짜로 만들면 들킴
- 47건이라고 했다가 다음 주에도 47건 = 죽은 도구
- **완화책:** 진짜 데이터 기반 자동 카운터

### ⚠️ 위험 3: 카톡 너무 그대로 만들면 차별점 모호
- "어차피 카톡인데 굳이 여기?"
- **완화책:** "카톡 + 진화" 명시 (검색·즐겨찾기·태그)

### ⚠️ 위험 4: 자료 정리 부담 그대로
- 라벨만 만든다고 자료가 정리되는 게 아님
- **완화책:** 비밀의 방 = 임시 보관소 → 검수 부담 줄어듦 (등급만 매기면 됨)

### ⚠️ 위험 5: 4팀 외 매니저·지점장 입점 요청 시
- "우리도 비밀의 방 같은 거 만들어달라"
- → **이게 v2.0 원수사 모델의 진짜 사례** (메모리 #13)
