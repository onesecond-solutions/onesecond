## 1. 출발점 — 원세컨드를 만든 가장 기본적인 불편함

```
[현장 설계사가 매일 겪는 것]

받는다 → 카톡방 10개+ / 단톡방 / 공지방 / 자료방
        → 푸시 문자 / 회사 공지 / 매니저 공지
        → 원수사 공지 / 상품 안내 / 인수 변경

쌓인다 → 스크롤 위로 위로 위로
        → "어제 본 그거 어디 갔지?"
        → "그 인수 기준 어느 방에 있었지?"

정작 필요할 때 → 고객 전화 중 / 상담 진행 중
              → 못 찾는다 → 멘트 끊긴다 → 신뢰 깨진다
```

### 3가지 불편함으로 분리

| # | 불편함 | 본질 |
|---|--------|------|
| 1 | Input Overload | 받는 양이 너무 많다 |
| 2 | Wrong Structure | 시간순으로 쌓여서 못 찾는다 |
| 3 | Context Mismatch | 받는 위치 ≠ 찾는 위치 ≠ 실행 위치 |

## 2. 원세컨드의 해결 모델 — "있다"의 신호

### 푸시의 정체성 전환

```
[기존 카톡 푸시]
"지금 봐라" 명령 → 안 보면 묻힘 → 압박감

[원세컨드 푸시]
"있다"는 신호 → 안 봐도 됨 → 나중에 들어가면 그대로 있음
```

### 푸시와 저장소의 분리

```
[카톡 구조]
푸시 = 메시지 = 저장소 (한 덩어리)

[원세컨드 구조]
푸시 = 신호 (알림만)
저장소 = 게시판/스크립트/MY SPACE (구조화)
   → 푸시 못 봐도 저장소에 그대로 있음
   → 시간순 + 주제별 + Stage별 다층 인덱스
```

### 모바일·웹 역할 분리

```
모바일 = 인지 채널 (푸시·짬 시간·가볍게)
웹·PC = 실행 채널 (상담 중·헤드셋·깊게)
   → 카톡과 결정적 차이: 역할이 분리됨
```

## 3. C영역 재정의 — 푸시 인박스

### B / D / C 역할 분리

| 영역 | 정체성 | 사용자 행동 |
|------|--------|------------|
| B | 능동 이동 | "어디로 갈까" 네비게이션 |
| D | 능동 작업 | "지금 하고 있는 일" 콘텐츠 |
| C | **수동 수신** (재정의) | "내가 모르는 사이 들어온 것" 인박스 |

### C영역 세분화 구조

```
[C영역 — 우측 패널 v1.1]

⚡ 빠른실행 (최상단 고정)
─────────────────────────
🟪 시스템 공지         [N]   ← 전산장애·점검·업데이트
🟦 회사 스텝 공지      [N]   ← 본사·지점 운영
🟧 매니저 공지         [N]   ← 팀/지점 매니저
🟩 원수사 공지         [N]   ← v2.0 진입 시 본격
🟨 4팀 Lab 알림        [N]   ← 4팀 멤버만 노출
─────────────────────────
⚙️ 알림 설정
[모두 확인] [전체 보기]
```

### 진화 경로

| 버전 | C영역 출처 |
|------|----------|
| v1.1 | 시스템 + 스텝 + 매니저 + 4팀 |
| v2.0 | + 원수사 (입점 본격) |
| v3.0 | + 스텝 세분화 + CRM 알림 |

## 4. 알림 5단 강도 분화

```
[알림 강도 5단계]

1) 모바일 푸시        외부 인지 (앱 닫혀도)
2) PC 토스트          내부 실시간 인지
3) 호버 프리뷰        능동 가벼운 들춰보기
4) C영역 배너         정리된 인박스
5) D영역 본문         본격 보기

→ 사용자가 원하는 깊이로 선택
→ 카톡은 1단계뿐
```

### 브랜드 정체성 연결

```
"1초 안에 꺼내 쓴다" (스크립트)
+ "1초 안에 들춰본다" (호버 프리뷰)
= 원세컨드 1초 철학의 알림 영역 확장
```

## 5. PC 토스트 — 카톡 회피 + 위치 분리

### 핵심 문제 인식

```
우하단 = 카톡·잔디·Outlook·Windows 알림 격전지
   → 원세컨드 토스트 우하단 시 시각 충돌
   → 사용자가 카톡으로 오인하고 놓침
```

### 긴급도별 위치 분리 (3단 결합)

| 긴급도 | 위치 | 강도 |
|--------|------|------|
| 🔴 긴급 | D영역 상단 띠배너 | 강제 인지 + 소리 |
| 🟡 일반 | C영역 상단 인라인 슬라이드 | 자연 인지 |
| ⚪ 참고 | A1 🔔 펄스 + 숫자 | 가장 약한 신호 |

### 시각 차별화 원칙

- 카톡과 한 번에 구별 가능해야 함
- 브라운 톤, 출처별 배너 색
- 원세컨드 로고 작게 표시
- A1 검색창 영역 침범 금지

## 6. 호버 프리뷰

### 동작 흐름

```
C영역 배너 → 마우스 hover 0.3~0.5초
         → 왼쪽으로 320px 패널 펼침
         → 미확인 5건 표시 (시간+제목+본문 1~2줄)
         → 항목 클릭 = D영역 본문
         → "전체 보기" = 게시판 이동
         → 마우스 떼고 0.5초 후 사라짐
```

### Lab(4팀 비밀의 방)과의 관계

```
Lab     = 메뉴 호버 (사이드바)         ← 어제 5/4 결정
C영역   = 알림 배너 호버                ← 오늘 5/5 결정
   → 적용 대상 다름, 충돌 없음
   → Lab에서 UX 검증 → C영역 본격 적용
   → 같은 호버 컴포넌트 공유 가능
```

## 7. 피로 제어 4층

| 층 | 제어 방식 | 구현 난이도 |
|----|----------|------------|
| 1 | 출처별 on/off (사용자 선택) | 🟢 |
| 2 | 긴급도 자동 분기 (🔴🟡⚪) | 🟢 |
| 3 | 묶음 알림 (Burst Throttling) | 🟡 |
| 4 | DND 시간대 (사용자 설정) | 🟢 |

### 원세컨드 차별화 — 상담 모드 자동 감지

```
스크립트 페이지에서 "상담 시작" → 토스트 자동 OFF
"상담 종료" → 자동 ON
   → 사용자가 매번 DND 켜고 끄지 않아도 됨
   → 카톡은 못 하는 컨텍스트 인식
```

## 8. 발신자(매니저·원수사) UX

### 공지 발행 시 강도 선택

```
🔴 긴급 → 강제 토스트 + 소리 (정말 긴급할 때만)
🟡 일반 → 토스트만 (기본값)
⚪ 참고 → 배지만 (참고용)
```

### 가이드 원칙

- 긴급도 기본값 = 🟡 일반
- 🔴 긴급은 의도적으로만 선택
- 가이드 문구: "긴급은 정말 긴급할 때만"
- 무분별한 🔴 사용 시 피로 폭격 = 본질 파괴

## 9. v2.0 게시판 진화 모델 (선행 통찰)

### 핵심

```
팀·지점 게시판의 상품/인수 카테고리
   = v2.0에서 원수사 출처의 "공통 디스플레이 창"으로 진화
   = 사용자 학습 부담 0
   = 게시판 구조 변경 없음, 운영 주체만 바뀜
```

### "함께 성장하는 요금제" 정당성

```
[원수사 입장에서 월 100만원의 가치]

원세컨드 입점:
   → 발행 위치 = 설계사 상담 중 사용 위치
   → 도달률 100%
   → C영역 미확인 [N] 자동 표시 = 확인율 정량 측정
   → DAU·Stage·열람 데이터 수집
   → 광고 매체 ROI 측정 가능

→ 월 100만원 = "도달이 보장된 발행 채널" 구독료
→ 원수사·설계사 양쪽 불편함이 같은 구조로 풀림
```
