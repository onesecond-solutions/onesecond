### C. 인앱 알림 Phase 3 — 웹푸시 (#398 + Edge Function + 트리거)
- **결정: 별도 sw-push.js 안 만들고 기존 `sw.js`에 push 핸들러 추가**(index.html이 이미 /sw.js 등록 중, 한 스코프 1 SW). `push`(알림 표시)·`notificationclick`(화면 열기·포커스) 추가, fetch/캐시 로직 무변경, CACHE_NAME만 v132로. 설계서의 "별도 파일" 안은 기존 sw.js 존재 모르고 쓴 것 → 정정.
- **구독**(app.html): `notifEnablePush()`(권한 요청→pushManager.subscribe→`push_subscriptions` upsert `on_conflict=endpoint`), VAPID 공개키 + base64 헬퍼. 알림 드롭다운에 버튼.
- **DB**: `push_subscriptions(id,user_id,endpoint unique,p256dh,auth,created_at)` + RLS `push_sub_own`(for all, 본인) 신버전 실행.
- **발송 Edge Function `send-push`**: VAPID 서명 웹푸시(npm:web-push), `x-cron-secret` 보안, 만료(404/410) 자동삭제. **CLI로 배포**(`npx supabase functions deploy send-push --project-ref pdnwgzneooyygfejrvbg --no-verify-jwt --use-api`, 팀장님 임시 PAT 발급→배포 후 폐기). 시크릿 `VAPID_PRIVATE_KEY`/`VAPID_SUBJECT` 등록.
- **발송 테스트**: 시크릿 없이 401(차단) / 포함 시 `{ok:true,total:1,sent:1}` + 기기 알림 도착 확인.

## 격차 / 학습 / 피드백
- **"구독" 단어 금지**(신규 피드백): 사용자가 무조건 유료로 오해. 무료 기능은 "알림 받기/설정". 진짜 결제만 "구독" 허용. 메모리 신설.
- **PAT 토큰 흐름**: 채팅 마스킹으로 첫 시도 값 안 보임 → 둘째 시도 평문 → Code가 직접 CLI 배포 → 팀장님 즉시 폐기. 정상 처리.
- **서비스워커 안전**: fetch 로직 무변경 + node --check 문법검사 후 머지. push 핸들러는 푸시 도착 시에만 동작 → 머지 자체로 사이트 영향 0.
- **별도 sw-push.js 함정 회피**: 기존 sw.js 존재 확인 후 거기에 추가(한 스코프 1 SW). 설계서 맹신 X = 코드 실측.
- **JSON 리터럴 CRLF 사고**: attachments에 `'[...]'::jsonb` → 0x0d 에러. `jsonb_build_array/object`로 해결.
- main 직접 커밋 0건(전부 feature 브랜치) — 잘 지킴. 팀장님 검수는 "머지→라이브 확인" 방식 채택(저위험 시각 UI).
