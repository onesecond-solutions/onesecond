### ③ RLS 3정책 + qna branch_id (#306 + DB)
- **라이브 검수에서 발견**: ga_member 계정 스크립트 저장 시 `new row violates row-level security policy for table "scripts"`.
- **진단 (실제 정책 SELECT로 확정)**: scripts는 INSERT 정책이 **admin(ALL)밖에 없음** → ga_member 차단. library는 `library_insert_own` 있어 정상. team_notices는 매니저 이상만. posts qna는 admin/seed만.
- **팀장님 결정**: 단체방·보험Q&A 모두 **설계사 작성 허용**.
- **DB RLS 3정책 신버전 실행** (additive, 본인 명의 scoped):
  - `scripts_insert_own` (WITH CHECK `auth.uid()::text = owner_id`)
  - `team_notices_insert_member` (scope=team_internal AND author_id=auth.uid AND 본인 team)
  - `posts_insert_qna_member` (board_type=qna AND auth.uid()::text=author_id AND branch_id=my_branch_id())
- **#306 코드**: `_saveMyspaceQna`에 branch_id 세팅(본인 지점 노출 + RLS 통과).
- **핵심 발견**: `os_user.id` = 로그인 응답 `data.user.id` = `auth.uid()` (db.js mergeUserProfile은 role/name만 덧붙임). 즉 owner_id 값 자체는 정상 → 문제는 정책 누락.
