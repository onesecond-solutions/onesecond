## 이번 세션 처리 (코드 변경 없음, 운영·상담)

### 1. OCR 색인 가속 (대표 DB 직접 실행)
- 구조: `pg_cron(jobid=5, ocr-batch-5min) */5분 → ocr-batch(미처리 N건) → ocr-extract(Gemini) → newsletters.full_text / myspace_files.search_text`.
- 빠른 처리법 = cron 주기 단축. 대표가 `SELECT cron.alter_job(5, schedule := '*/1 * * * *');` 실행 → **1분 주기로 가속**(밤새 자동 색인). 끝나면 `*/5` 원복.
- 과금 = 처리 파일수 비례(주기 무관, 비용 동일). PDF·이미지만 Gemini 과금, docx·office·18MB+ skip. rate limit(429)은 다음 틱 자동 재시도.
- **OCR은 Supabase 서버(클라우드)에서 도므로 대표 PC 꺼도 자동 진행.**

### 2. 자료실 10GB 대량 업로드 운영 상담
- 대표가 지점(branch) 자료실에 폴더 통째 업로드 → 일부 실패.
- 결과: 오늘 저장 성공 = branch 3,386 + team 25 + personal 2 = 약 3,413개. **실패 1,159건**(화면 빨간 목록).
- 실패 원인 진단: ① PC 절전/슬립(자리 비움 끊김) ② Storage 단일파일 50MB 기본 한도 ③ **한꺼번에 ~4,545개 동시 업로드 과부하**(브라우저·서버 throttle). 파일 키는 ASCII(uuid) 변환이라 한글파일명 400은 아님.
- 빨간 실패 목록 = 브라우저 화면 임시표시(새로고침/내일 사라짐). 실패=애초 저장 안 됨 → 자료실/DB 영향 0, orphan 없음.
- **대표 결정: 실패 1,159건은 "지난 자료라 버림"** → 재업로드 안 함. 추가 정리·soft delete 불필요. 그냥 새로고침으로 종료.
- (재업로드 시 권장이었던 것: 절전 끄기 + 크기한도 상향 + 폴더 단위로 나눠 올리기 — 이번엔 미적용, 버리기로 결정)
