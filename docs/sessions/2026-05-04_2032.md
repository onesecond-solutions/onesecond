## 학습 raw (영구 메모리 후보)

### 1. RPC PostgREST overhead 본질 변동성 (D-2 별 트랙 #3 청산 후 발견)

- RPC `get_stage_distribution()` 적용 후 P3 = cold 1143ms / warm 208~800ms (avg ~440ms)
- baseline P1 (243~1022ms) 대비 단축 효과 미미 (-14~27%)
- **본질 = PostgREST overhead + HTTP 라운드트립** (DB 본체 ~5~10ms 추정)
- "9 query → 1 query 변환 + payload 축소"는 패턴 표준 효과만 큼. 라운드트립 자체 단축은 Edge Function 또는 cache 전략 영역
- D-3 P1·P2 회귀 (warm 250~544ms)도 동일 패턴 — 임계 ≥1초 미달성으로 별 트랙 격상 불필요 청산
- 별 트랙 분석: `docs/specs/admin_v2_p3_postgrest_analysis.md` (4가설 — EXPLAIN ANALYZE / Sentry SDK / connection pool / Edge Function)

### 2. mock + v2.0 대기 패턴 표준화 (D-3 J-2 (b) + D-4 K-1 (c))
