## Q1. 진단 ①·② 사실 확인 — 둘 다 사실입니다

**진단 ① (좌↔우 선택 표시 없음) = 사실.**
- 결과 카드 `.sores`는 `onclick="_searchOpen(table,id)"`만 있고 **선택(active) 클래스·상태가 전혀 없음**(app.html:9577). CSS에도 `.sores.on`류 없음(826~834). 클릭해도 좌측은 시각 변화 0 → 어느 항목을 보는지 안 보임.

**진단 ② (OCR raw 그대로 노출) = 사실.**
- 패널 본문 = `_searchOpen`의 `bodyTxt`(9710)를 `_sMark`(검색어 하이라이트)만 입혀 출력. **후가공 0.**
- 깨짐의 진짜 출처 = 표시 단계가 아니라 **추출 단계**. `ocr-extract`의 SYSTEM_PROMPT가 "원문 그대로, 요약·해석·생략 절대 금지"(index.ts:62~69) + `temperature:0` → 카드/표의 붙은 디자인 텍스트("유사암과소액암을제외한모든암")를 **띄어쓰기 없이 충실히** 가져옴. 즉 의도된 "원문 충실"의 부작용.

→ 기획팀장 처방 ②의 핵심 통찰(**"조판 ≠ 본문 품질"**, 클라 후처리 비권장) 코드 관점에서 **전적으로 동의**.

## Q2. 본문 텍스트 저장 구조 현황

- **OCR 결과 = raw 한 컬럼만.** 정제본 별도 컬럼 없음(`clean_text`/`refined_text`/`display_text` 코드 전수 0건).
  - 자료실 = `myspace_files.search_text`
  - 소식지 = `newsletters.full_text`
  - 채굴 지식 = `knowledge_entries.body`
- **패널이 읽는 필드**(`_searchOpen` bodyTxt 우선순위, 9710):
  `full_text || content || search_text || script_text || body || description || memo_text`
  → 자료=`search_text`, 소식지=`full_text`, 지식=`body`.
- ⚠️ **핵심 제약:** `search_text`는 **trgm 검색 인덱스 + 패널 표시 이중 용도**(#822 GIN 인덱스가 이 컬럼에 붙어있음). 표시를 정제하면 검색 대상도 함께 바뀐다 — 분리 설계의 분기점.

## Q3. 처방 ②(B안) 기술 영향 범위

- **삽입 지점:** `ocr-extract`(추출) 또는 `ocr-batch`(적재) 단계. 클라이언트 아님(기획팀장 판단 맞음).
- **DDL 규모(컬럼 분리 시):** `myspace_files`·`newsletters`·`knowledge_entries` 3테이블에 정제 컬럼 추가 → 검색은 raw, 표시는 정제본. trgm 인덱스도 정책에 따라 추가.
- **소급 정제:** 원하면 기존 자료실 OCR분 + 소식지 461 + 지식 914 재처리. **단 raw 텍스트가 이미 있으니 이미지 재OCR 불필요** → `raw 텍스트 → LLM 정제`만(이미지 전송 0, 저렴·빠름). `ocr-batch` 재실행 패턴 그대로.
- **검색 영향:** 정제본으로 노출하면 trgm 인덱스를 정제 컬럼 기준으로. 띄어쓰기가 생기면 오히려 검색 정확도↑(현재 raw는 띄어쓰기 없어 부분 매칭 누락 가능).

## Q4. 처방 ①(active state) 난이도 — 낮음, 기존 패턴 재활용 가능

- **난이도 하.** 이미 `.sv-tab.on`(소스 탭)·스코프 색 등 `.on` 토글 패턴이 코드에 있음 — 동일 방식.
- **구현:** ① 카드 클릭 시 선택 키(`table:id`)를 `window._svOpenKey`에 기록(이미 있는 `_svPanelHtml` 보존 패턴과 형제) ② `_svRender`에서 각 카드 렌더 시 키 일치하면 `.sores.on` 부여 ③ CSS `.sores.on{border-left:3px solid var(--ac); background:색틴트}` ④ 패널 헤더 출처 칩(이미 `sv-d-chip` 구현됨)과 좌측 태그 색을 같은 언어로.
- 탭 전환·더보기 후에도 선택 유지(`_svOpenKey` 보존). **반나절 내 가능.**

## Q5. ★ 더 나은 해결책 (코드/스택 관점)

기획팀장 B안(채굴 1회 정제) 방향에 동의하되, **구현은 두 갈래로 나뉘고 권고가 갈립니다.**

| | 갈래 B-1 (권고) | 갈래 B-2 |
|---|---|---|
| 방식 | `ocr-extract` 프롬프트를 **"내용 가감 0 + 한국어 띄어쓰기·문단만 복원"**으로 강화 → `search_text` 자체가 정제본 | 정제 전용 컬럼(`display_text`) 신설 → 검색=raw, 표시=정제 |
| DDL | **0** (컬럼 유지) | 3테이블 컬럼 추가 + 인덱스 |
| 검색 | 정제본 검색(띄어쓰기 생겨 정확도↑) | raw 검색 유지 |
| raw 원문 보존 | 안 됨(정제본으로 대체) | 됨(증거 보존) |
| 소급 | `ocr-batch` 재처리(raw→정제, 저렴) | 동일 + 컬럼 백필 |

- **클라 후처리(표시 시점 띄어쓰기 복원) = 비권장 확정** (기획팀장 단서와 동일: 한국어 분절 부정확·매 표시 비용).
- **코드 관점 권고 1줄:** **B-1 채택** — `search_text`가 검색·표시 이중용도라 컬럼을 쪼개면 trgm 인덱스·코드 양쪽이 늘고, 정제본은 검색 정확도까지 올린다. **단 "raw 원문 증거 보존"이 정책상 필수면 B-2.**
  - 안전장치: 정제 프롬프트는 `temperature:0` + "**내용·숫자·용어 변경 금지, 띄어쓰기/줄바꿈만 복원**" 명시 → 요약·왜곡 위험 차단.
- **단계적 적용안(권고 순서):**
  1. **active state(Q4)** 먼저 — 코드만, DB 0, 즉시 체감(좌↔우 결속). 독립 PR.
  2. **OCR 정제(B-1)** — `ocr-extract` 프롬프트 강화 PR + 검수팀 재배포 → 신규분 자동 정제. 기존분은 `ocr-batch` 재처리(밤샘 cron, 과금=파일수 비례).
  3. (선택) raw 보존 정책 결정되면 B-2로 승급.

## 회신 요약 (대표님 결재용)

- 진단 ①·② **둘 다 사실**. ②의 깨짐은 표시가 아니라 **OCR 추출 프롬프트("원문 그대로")의 부작용**.
- 정제본 컬럼 **없음**. `search_text`가 **검색+표시 이중용도**(분리 여부가 결정 포인트).
- **권고:** ① active state는 쉬우니 먼저(코드만) → ② OCR는 **B-1(추출 프롬프트에 띄어쓰기 복원 추가, DDL 0)**, raw 증거 보존이 필수일 때만 B-2(컬럼 분리).
- 다음 = 대표님 결재 → 작업지시서 별도 발행 후 착수.
