# 지식 페이지 — 셸 통합 가독성 스펙

**작성: 기획팀장(Claude Web) → 총괄팀장(Code)**
**작성일: 2026-07-22**
**대상: app.html 셸(D 중앙 영역)에 얹히는 모든 지식 페이지**

---

## 0. 이 문서가 나온 이유

라이브(app.html)에 얹힌 간병보험 페이지의 본문이 넓게 퍼져 가독성이 떨어졌습니다.
**총괄 구현이 틀린 게 아닙니다.** 제(기획팀장)가 셸 통합 시 본문 폭 제어 스펙을 주지 않아,
단독 HTML에서 걸려 있던 `max-width`가 셸 안에서 상속되지 않은 것입니다.
셸의 콘텐츠 컨테이너(`.viewer` 등)에는 폭 제한이 없어, 지식 페이지 본문이
중앙 영역 전체(최대 1500px 웹)까지 늘어납니다. 한 줄이 80자를 넘어가면서
눈이 좌우로 왕복하는 거리가 길어지는 게 실제 원인입니다.

---

## 1. 확정된 폭 기준 (팀장님 결정)

| 대상 | max-width | 근거 |
|---|---|---|
| **본문 (읽는 텍스트)** | **710px** | 한 줄 40~45자. 팀장님 확정값 |
| **표·다이어그램 등 넓은 요소** | **760px** | 필요한 경우에만 예외적으로 |

이전 v4에서 쓴 768px보다 살짝 좁힙니다. 팀장님이 라이브에서 직접 확인한 값이 710/760입니다.

---

## 2. 총괄이 적용할 방법

### 2-1. 지식 페이지 콘텐츠를 폭 제한 래퍼로 감싼다

셸의 콘텐츠 영역(`.viewer` 또는 지식 페이지가 마운트되는 컨테이너) 안에서,
**지식 페이지 루트에 폭 제한 클래스를 건다.** 셸 자체는 건드리지 않는다.

```css
/* 지식 페이지 전용 — 셸에 얹혔을 때도 항상 적용 */
.knw{
  max-width: 710px;      /* 본문 기본 폭 */
  margin: 0 auto;        /* 중앙 정렬 */
  padding: 0 20px;       /* 모바일 좌우 여백 */
}
/* 표·다이어그램만 예외적으로 넓게 */
.knw .wide{
  max-width: 760px;
  margin-left: auto;
  margin-right: auto;
}
```

핵심: **단독 페이지의 body에 max-width를 걸면 셸 안에서 무효가 된다.**
반드시 지식 페이지 **루트 요소(.knw)** 에 걸어야 셸/단독 양쪽에서 동일하게 동작한다.

### 2-2. 본문 타이포 (v3 가독성 표준 유지)

```css
.knw{ font-size:16px; line-height:1.8; word-break:keep-all; }
.knw p{ margin:0 0 1.1em; }
.knw .caption{ font-size:13px; }   /* 13px대는 캡션 전용 */
```

- 본문 16px (최소 15px). **13px대는 캡션·부가설명 전용**
- 행간 1.8
- `word-break:keep-all` — 한글 단어 중간 끊김 방지

---

## 3. 셸 폭에 따른 반응 (양쪽 사이드바 있는 구조)

셸은 좌측(B) 사이드바 + 중앙(D) + 우측(C) 목차/CTA 구조입니다.
지식 페이지 본문은 **셸 폭과 무관하게 항상 710px를 유지**합니다.

```
┌─ B ─┬────────────── D (viewer) ──────────────┬─ C ─┐
│ 사이드 │   ← 여백 →  [본문 710px]  ← 여백 →      │ 목차 │
└─────┴────────────────────────────────────┴─────┘
```

- D 영역이 아무리 넓어도 본문은 710px, 남는 공간은 좌우 여백으로 비운다
- C(우측 목차·CTA)는 셸이 이미 제공하거나, 지식 페이지가 1180px 이상에서만 자체 렌더
- **표는 760px까지 확장 가능** (`.wide` 클래스)

---

## 4. 반드시 피할 것

| 하지 말 것 | 이유 |
|---|---|
| `body{max-width}` 로 폭 제어 | 셸 안에서는 body가 최상위가 아니라 무효 |
| 표에 `min-width` 강제 | 모바일에서 가로 스크롤 발생 (반응형 표준 위반) |
| 본문을 D 영역 100%로 렌더 | 지금 문제의 원인. 한 줄 80자+ |
| 13px를 본문에 사용 | 캡션 전용. 본문은 16px |

---

## 5. 검증 방법

총괄이 적용 후, 아래를 확인:

1. **단독으로 열었을 때** 본문 710px 유지되는가
2. **app.html 셸에 얹혔을 때** 동일하게 710px 유지되는가 ← 이게 핵심
3. 표가 760px까지만 넓어지고 모바일에서 가로 스크롤 없는가
4. 셸을 collapsed(사이드바 접기) 했을 때도 본문 폭이 안 흔들리는가

라이브에서 한 줄 글자 수를 세어 **40~45자**면 통과입니다.

## 부록 — 총괄팀장(Code) 대조 결과 (2026-07-22)

기획팀장 스펙을 실제 코드·아키텍처와 대조한 결과. **스펙의 폭 결론(710/760)은 정확하고 이미 반영됨.** 다만 아래 3가지가 실제와 다르거나 스펙에 빠져 있어 기록한다.

### A. 통합 방식 = iframe (스펙 전제인 "직접 마운트"와 다름)
- 스펙 §0·§2-1은 지식 페이지를 셸 `.viewer`에 **직접 마운트** → `body{max-width}` 무효 → 루트 `.knw` 클래스 필요, 라는 전제.
- **실제:** 지식 페이지(간병 등)는 **iframe**(`#caregiver-frame` 등)으로 로드된다. 각 페이지가 **독립 문서**라 페이지 내부 `max-width`가 정상 작동한다 → 그래서 V3 710px가 iframe 안에서 이미 먹는다. "셸 안에서 max-width 무효"는 iframe 구조엔 해당하지 않는다.
- 결론: **iframe 유지 시** 스펙의 `.knw` 루트 클래스는 불필요(페이지 내부 폭 제어로 충분, 이미 작동). `.knw`/공통 knowledge.css 방식은 **직접 마운트로 전환할 때** 유효. (현재 결정: iframe 유지 — 리스크 최소·이미 작동.)
- (참고: `.viewer`·`tokens-v2.css`는 앱에 실재하나 iframe 지식 페이지의 마운트 지점이 아님. `knowledge.css`는 미존재=신설 필요 시점은 직접 마운트 전환 시.)

### B. 게이트 단계 (스펙엔 없음)
- V3 710px 가독성은 현재 **임태성 게이트 전용**(`html.v3`, `window.parent._canSeeXfile()`). **일반·비로그인 사용자는 아직 옛 1240px를 본다** — 기획팀장이 관찰한 "본문이 넓게 퍼짐"은 이 게이트 밖 화면이 실제로 그렇다(정확한 관찰). 대표 지시 "게이트 먼저 → 확인 후 전면 개방" 단계라 의도된 상태.

### C. 제목 위계 (스펙 §2-2에 빠짐 — 실제 가독성 원인)
- 스펙 §2-2는 본문 타이포만 규정하고 h1/h2/h3 크기를 정하지 않는다. 그 결과 옛 v2의 작은 제목(h2 16.5px ≈ 본문 16px)이 그대로 남아 **위계가 납작**해져, 게이트 710px에서도 가독성이 떨어졌다(총괄 진단, 대표 확인).
- **조치(2026-07-22):** `html.v3` 스코프로 기획팀장 v3 디자인의 제목 위계 복원 — 데스크톱 h1 27 · h2.sec 20 · h3 19 · 본문 16. 폭·게이트·회귀0 유지.
- **차기 스펙 반영 권고:** 지식 페이지 표준에 **제목 타입스케일**을 명시할 것(본문 16 기준 h3 18~19 / h2 20 / h1 27, 모바일 비례 축소).
