# 상담관리 리스트 뷰 — 통합 설계 v1 (영업노트 자산 재사용)

> 상태: 설계(대표 승인 대기) · 작성 2026-07-10
> 산출물 성격 = **기획팀장(Claude Web) "상담관리 리스트 뷰 A안" 지시서를 `sales_customers` 재사용으로 정합화한 재작성 설계** (대표 확정 2026-07-10).
> 선행 확정 설계: [`docs/specs/consult_customer_form_v1.md`](./consult_customer_form_v1.md) — 폼·OCR·단계·인라인 흐름 = 영업노트 정본. 본 문서는 그 **리스트(목록) 뷰** 짝 문서다.
> 오늘 범위 = 이 문서 1개. **제품 코드 수정 0 · DB write 0 · DDL 0 · 커밋 0.** 아래 실측 근거는 라이브 코드(app.html)에서 직접 추출(추정 아님, 파일:라인 명시).

## 0. 정합화 요지 (한 문단)

기획팀장 A안 원안은 상담관리 리스트를 **신규 `customers` 테이블 + 7단계 STAGES**로 설계했다. 그러나 선행 확정 설계(form_v1 §1)가 이를 뒤집어 **기존 영업노트 자산(`sales_customers`/`sales_consultations`) 재사용**으로 확정했고(신규 테이블 금지·고객 원장 이원화 금지), 대표가 **A안 = sales_customers 재사용 정합화**를 확정했다. **결정적 실측 발견:** 기획팀장이 설계한 리스트 뷰 골격(좌측 목록 + 상태 필터칩 카운트 + 카드 클릭 → 우측 상세/폼 + 이름·전화 검색 + 소프트삭제 + owner 격리 + 빈상태)은 **신규로 만들 것이 아니라 이미 영업노트 `#v-salesnote`에 통째로 구현돼 라이브 운영 중**이다(임태성 게이트). 따라서 본 정합화의 본질 = **"신규 리스트 화면 신설"이 아니라 "이미 존재하는 영업노트 리스트 뷰를 업무노트 '상담관리' 유형 진입점으로 연결하고, 임태성 게이트에서 단계적으로 개방"**하는 것이다. **DB 변경 0.**

## 1. 기획팀장 A안 골격 ↔ 영업노트 실측 대응표 (셸 재사용 지점)

기획팀장 지시서의 리스트 골격 요소별로, 영업노트에 이미 존재하는 구현을 실측 매핑했다. **"신규 구현"이 아니라 "재사용/연결" 판정이 핵심.**

| A안 리스트 골격 요소 | 영업노트 실측 구현 (app.html) | 판정 |
|---|---|---|
| 리스트 셸(좌 목록 / 우 상세 2단 레이아웃) | `.sn-layout` = `#sn-list`(좌) + `#sn-detail`(우), app.html:2792~2795 | **이미 존재 — 재사용** |
| 이름·전화 상단 검색 | `#sn-q` placeholder "이름·전화 검색" + `_snMatch` 필터, 2790·3926~3929 | **이미 존재 — 재사용** |
| 상태 필터칩 + 칩별 카운트 | `_snRenderChips()` = `['전체']+_SN_STAGES` 각 칩에 `<b>n</b>` 카운트, 3915~3924 | **이미 존재 — 재사용** |
| 카드(행) 클릭 → 우측 상세 로드 | `_snSel_(i)` → `#sn-detail`에 이름·전화·상태배지·상담이력·인라인 폼 진입버튼 렌더, 3939~3957 | **이미 존재 — 재사용** |
| 카드 클릭 → 우측 인라인 고객 폼 | `_snOpenForm(i)` → `_snFormHtml` 우측 인라인 렌더(신규/수정 겸용), 4076~4095 | **이미 존재 — 재사용(form_v1 §3·§6)** |
| 소프트삭제(deleted_at) | `_snDelete(i)` = `deleted_at` PATCH(휴지통식, 목록에서만 숨김), 4124~4130 | **이미 존재 — 재사용** |
| 목록 조회 시 삭제행 제외 | fetch `?deleted_at=is.null`, 3908 | **이미 존재 — 재사용** |
| RLS user_id(owner) 격리 | 조회는 RLS가 `owner_id=auth.uid()` 자동 필터(주석 3890·3908), INSERT `owner_id=_snOwnerId()`(4120) | **이미 존재 — 재사용** |
| 빈상태 UX | "왼쪽에서 고객을 선택하세요"(2794)·"검색 결과 없음"(3935)·"상담 기록 없음"(3963)·"불러오기 실패"(3912) | **이미 존재 — 재사용** |
| '상담관리' 칩 = 개인 스코프 한정 | 영업노트 자체가 개인(owner)·비공개 화면(`rm-note` "🔒 비공개", 2787) | **개념 일치 — §4에서 스코프 규칙 확정** |

**결론:** A안 리스트 골격 10요소 중 **9요소가 영업노트에 이미 구현·라이브 운영 중**, 1요소('상담관리' 칩 스코프)만 연결 시 확정. **리스트 셸을 새로 만들 필요가 없다.**

> ⚠️ 지시서 원안의 "신규 `customers` 리스트를 별도 화면으로 구축"은 **폐기.** 이유 = 동일 리스트가 `#v-salesnote`에 존재하며, 신규 화면은 (a) 고객 원장 이원화(form_v1 §1), (b) 리스트 UI 중복 구현 = 회귀 반경만 키움.

## 2. 지시서 필드 ↔ `sales_customers` 실제 컬럼 매핑표

리스트 카드/필터가 참조하는 데이터 계약을, 지시서 필드명 → 실제 컬럼으로 교체. 실사용 컬럼은 app.html·form_v1 §2에서 실측.

**실측 컬럼(app.html:3908 select · 4106~4120 write · 4087 profile fetch):**
- `sales_customers`: `id · name · phone · phone_raw · birth_date · gender · channel · status · owner_id · source_ref · deleted_at · created_at · updated_at · profile(jsonb)`
- `profile` jsonb 실사용 키: `note_html`(리치 상담 메모) · `customer_type`(유입 플랫폼) · `age`

| 지시서(A안) 리스트 필드 | 실제 매핑 | 리스트에서 쓰임 | 판정 |
|---|---|---|---|
| `name` (고객명, 카드 제목) | `sales_customers.name` | 카드 제목(`.nm`, 3933) | 기존 존재 |
| `attr` (속성) | `profile.attr`(미사용 키, 흡수 가능) | (현재 리스트 카드 미표시) | profile jsonb 수용 / **의미 확인 필요(§6)** |
| `stage` (상태, 필터칩·배지) | `sales_customers.status` | 필터칩 카운트(3920)·카드 상태배지(3944) | 기존 존재 (단계 정본 = §3) |
| `phone` (연락처, 카드 서브) | `sales_customers.phone` / `phone_raw` | 카드 서브(`.ph`, 3933)·검색 대상(3928) | 기존 존재 |
| `updated_at` (정렬/최근) | `sales_customers.updated_at` | 정렬 후보(현재는 `name.asc` 정렬, 3908) | 기존 존재 |
| `deleted_at` (소프트삭제 제외) | `sales_customers.deleted_at` | 목록 조회 `is.null` 필터(3908) | 기존 존재 |
| `owner`(소유자, 격리) | `sales_customers.owner_id` | RLS 자동 필터(격리 레이어) | 기존 존재 |
| `memo`(상담 내용) | `sales_consultations.memo`(회차) + `profile.note_html`(리치) | 우측 상세 상담이력(3961~3968)·폼 리치(4028) | 기존 존재 |

**`sales_consultations` 실측 컬럼(3961 select · 3986 write):** `id · customer_id · owner_id · consulted_at · channel · memo · created_at`.

**진짜 부족(ALTER 필요) 필드 = 0.** 리스트 뷰가 표시·필터·정렬에 쓰는 값(name·status·phone·updated_at·deleted_at)은 전부 기존 실컬럼. `attr` 하나만 의미 미확정이나 표시 대상이 아니면 흡수 불필요(§6).

## 3. 단계 상수 정본 판정 (7단계 vs 8단계)

**정본 = 영업노트 실측 8단계.** 근거는 코드 실측 + 실데이터 운영.

- 영업노트 실측: `var _SN_STAGES=['신규DB','부재','예약','진행중','제안서발송','보류','클로징','청약완료'];` (app.html:4002, 주석 "노션 상담 단계 2026-06-29 대표 확정"). 8단계.
- 이 8단계는 (a) 필터칩 카운트(`_snRenderChips`, 3917), (b) 폼 영업상태 select(`_snFormHtml`, 4007), (c) 저장 기본값 `'신규DB'`(4108) 세 곳의 **단일 출처**로 이미 라이브 운영 중이다.
- 임태성 실데이터 **2,583명**의 `status` 컬럼이 이 8개 값으로 이미 저장돼 있다(form_v1 §5). 지시서 7단계 라벨(부재·첫콜·상담중·제안서·클로징·청약완료·관리종결)로 바꾸면 **저장값 마이그레이션이 필요 → 회귀 반경 큼.**

**판정: 기존 8단계를 정본으로 유지.** (form_v1 §3 총괄팀장 의견과 동일 결론)
- 완전 일치: 부재·클로징·청약완료(3). 근사(라벨만 상이): 진행중↔상담중, 제안서발송↔제안서(2).
- 지시서에만 있음: 첫콜·관리종결(2) — 실제 필요하면 **기존 배열에 값 추가로 흡수**(예: 종결 상태 '관리종결' append). 마이그레이션 없이 확장 가능. 기존 값 라벨 변경은 하지 않는다.
- 기존에만 있음: 신규DB·예약·보류(3) — 실데이터가 쓰는 값이므로 유지.

**상수화 방향(오늘 실행 아님):** 현재 `_SN_STAGES`는 app.html 인라인 상수. 향후 `js/constants/stages.js`로 분리해 영업노트·상담관리·(미래)통계가 한 정본 배열을 공유. 리네임이 아니라 값의 단일 출처화라 회귀 위험 낮음.

## 4. '상담관리' 칩 스코프 규칙 — 개인 한정 (공유 스코프 노출 금지)

- **상담관리 칩은 개인(마이) 스코프에서만 노출한다.** 팀·지점 등 공유 스코프에 노출 금지.
- 이유: `sales_customers`는 고객 실명·연락처·생년월일 등 **개인정보(PII)**를 담는다. 공유 스코프에 칩을 띄우면 "공유되는 데이터"로 오인될 위험 + 흡수 오해(메모리 `feedback_no_platform_asset_absorption_copy`) 소지. 실제 데이터는 owner_id RLS로 개인 격리(§5)되므로, **UI 진입점도 개인 스코프에만 두어 개념 일관성**을 유지한다.
- 영업노트 자체가 이미 개인·비공개 화면으로 라이브(`rm-note` "🔒 비공개 — 본인 영업 데이터 관리 전용 화면", app.html:2787)이므로, '상담관리' 유형 = 이 비공개 화면으로의 **개인 스코프 연결 라벨**이다.

**라이브 용어 정합(live_term_glossary):**
- **영업노트** = `#v-salesnote` = 라이브 표시 용어(glossary §1). 소통·화면 문구는 "영업노트" 사용.
- **상담관리** = form_v1가 도입한 **업무노트 유형 라벨**(glossary 미등재 신규 용어). 리스트 뷰 문구·칩 라벨은 "상담관리"로 쓰되, 코드가 가리키는 실체는 `#v-salesnote`(안정키)임을 매핑 주석으로 남긴다(표시명↔안정키 분리 원칙, glossary §3).
- ⚠️ 옛 용어("영업 CRM"·"고객 원장"·"customers" 등) 화면 노출 금지.

## 5. 게이트·격리 재사용 (영업노트 기존 방식)

리스트 뷰는 영업노트의 게이트·격리를 **그대로 재사용**한다. 신규 정책 신설 0.

- **UI 노출 게이트:** `_canSeeSalesNote()` = user_id `98c5f4f9-10c1-4ee1-a656-5c2ca63239fd`(임태성) 전용 + CSS `body.is-uat-salesnote`(app.html:3891·4200). 리스트/상담관리도 **1단계 = 임태성 게이트 안에서만** 먼저 배선·검증.
- **데이터 소유·격리 레이어(항상 유지):** `owner_id` RLS. 조회는 `owner_id=auth.uid()` 자동 필터, INSERT는 `owner_id=_snOwnerId()`(app.html:3908 주석·4120). UI 게이트를 넓혀도 데이터 격리는 1인치도 안 넓어진다(form_v1 §6-1 레이어 분리 원칙 재확인).
- **소프트삭제:** `deleted_at` PATCH(app.html:4124~4130), 목록 조회 `deleted_at=is.null`(3908) — 재사용.
- **개방 롤아웃:** form_v1 §6-1의 3단계(1: 임태성 → 2: 더원지점 4팀 team_id 게이트 → 3: 전체)를 그대로 따르며, 각 단계 확대는 **form_v1 §5 owner_id 격리 실측 통과 + 대표 승인** 후에만. 리스트 뷰는 별도 개방 정책을 만들지 않는다(폼과 동일 게이트 공유).

## 6. DB 변경 필요 여부 판정 ⭐

**판정: DB 변경 = 0 (기존 컬럼/jsonb 재사용으로 충분).**

- 리스트 뷰가 쓰는 모든 값(name·status·phone·phone_raw·updated_at·deleted_at·owner_id)은 `sales_customers` **기존 실컬럼**이다(§2). 상담이력은 `sales_consultations` 기존 컬럼.
- 상태 8단계는 이미 `status` 컬럼 + `_SN_STAGES` 상수로 운영 중(§3). 신규 컬럼/테이블 불필요.
- 지시서 잔여 필드(attr·email·zip·addr 등)는 리스트 표시 대상이 아니거나, 필요 시 form_v1 §2대로 `profile` jsonb 키로 흡수 가능(ALTER 0). **실데이터 2,583행에 대한 스키마 변경 위험 0.**

**⚠️ 새 컬럼/테이블이 꼭 필요해지는 조건(현재는 해당 없음 — 발생 시 진행 중단·별도 승인):**
1. `attr`(속성)의 의미가 확정돼 **리스트 필터·정렬·통합검색 대상**으로 승격해야 할 때 → jsonb는 인덱싱·정렬이 약해 컬럼 승격(ALTER) 검토. 단순 저장·표시면 profile jsonb로 충분.
2. 주소/이메일을 **정렬·필터·통합검색** 대상으로 강하게 걸 계획일 때(§2·form_v1 §2와 동일 조건).
> 위 조건에 해당하면 **임의로 ALTER 진행하지 말고** 별도 승인 트랙으로 플래그하고 중단. 현재 설계는 ALTER 0 유지.

## 7. 회귀 위험 (영업노트 기존 화면 무영향)

- **오늘 산출물 = 문서 1개(DB write 0 · 코드 수정 0)** → 라이브 회귀 물리적 0.
- 실제 위험은 **화면 배선 착수 시점**에 발생:
  - 영업노트는 실데이터 **2,583명 / 상담 2,621건**을 임태성 게이트 안에서 운영 중. 리스트 셸(`#v-salesnote` `sn-*`)을 재사용하므로 **기존 함수·DOM id·CSS 클래스(`sn-layout`·`sn-list`·`sn-detail`·`_snSel_`·`_snRenderChips` 등)를 변경하면 라이브 영업노트가 즉시 영향**받는다.
  - 원칙: **기존 `sn-*` 자산은 무변경 재사용, '상담관리' 진입점(업무노트 유형 선택 → 영업노트 호출)만 추가**한다. 기존 셸 로직 수정 최소화.
  - 배선은 form_v1 §6대로 **1단계(임태성 게이트) 안에서 Deploy Preview 검수 + 배포 7단계(대표 승인 후 AI팀 머지)** 통과 후 확대.
- **glossary 안정키 회귀 주의:** 코드 식별자(`_SN_STAGES`·`sn-*`·`_canSeeSalesNote`)는 안정키. 표시명만 "상담관리/영업노트"로 통일하고 **키는 리네임하지 않는다**(glossary §3, #1024~#1026 회귀 전례).
