# onesecond 지식 분류 체계 (taxonomy) v1 — 보완안

> 작성: 2026-06-18. 기획팀장(Web) 보완안 + 총괄팀장(Code) 현 데이터 매핑 분석.
> 상태: **대표님 결재 대기.** 결재 후 → 20건 샘플 매핑 → 미세 수정 → 914 전체 적용.
> 원칙: **"화면 구조 = 채굴 구조."** 한 지식 entry = 아래 6좌표. 모든 채굴이 이 그릇에 태깅 → 검색 탭에 정확히 꽂힘.

## 6축 정의

### 축 1 — 소스 (1차 탭 · 단일값)
`보험지식 / 자료 / 소식지 / Q&A / 스크립트 / 공지` (전체 = 합집합)
- 채굴 산출물 = 항상 **보험지식** (자료·소식지·Q&A에서 캐낸 정제층, 근거로 원천을 가리킴)
- 라이브 검색 탭 라벨: 지식 / 자료 / 소식지 / 보험Q&A / 스크립트 / 공지 (사용자 말, 내부 시스템명 노출 금지)

### 축 2 — 계열 (2차 필터 · 다중값) ← 본체
| 계열 | 정의 |
|---|---|
| 상품 | 원수사 상품 정보·특징·변경 (무엇을 파나) |
| 인수 | 가입 가능 여부·고지·인수/간편심사 기준 (이 고객 되나) — 심사 포함 |
| 보장 | 담보·특약·지급 조건 (뭘 보장하나) |
| 비교 | 회사·상품 간 보험료·보장 비교 (어느 게 나은가) — 중립=공정비교 핵심 |
| 상담 | 설득·반론·설명 멘트 (어떻게 말하나) |

### 축 3 — 교차태그 (다중값 · 계열 가로지름)
- 질병: 고혈압·당뇨·갑상선·용종·암 …
- 회사: 원수사명 (비교·상품 계열에서 특히)
- 상품군: 암보험·종합·운전자·치아·실손·종신·정기·유병자·간편심사 …

### 축 4 — 공간 범위 (단일값)
`개인 / 팀 / 지점 / 공통` — 채굴(보편 지식) 기본=공통, 지점 고유=지점. 개인·팀=사용자/개인 자료실 OCR 채굴 진입 시(미래).

### 축 5 — 출처 (메타 · 근거)
`소식지 / 자료 / 원문파일 / 작성자 / 등록일` + 근거 유형(약관·법령·상품설명서·공문·사례). **근거 1개 이상 없는 지식 = 미승인**(지어낸 지식 차단).

### 축 6 — 노출 상태 (단일값)
`ai_draft → 검수중 → approved`. approved만 검색 노출(현 RLS 일치).

## 매핑 규칙
자료 1건당: ①소스=보험지식 ②계열 ≥1 ③교차태그(가능한 만큼) ④공간(기본 공통) ⑤근거 ≥1 ⑥상태=ai_draft. 분류 안 되면 **"미분류" 보류**(억지 태깅 금지).

## 적용 순서 (전체 전 샘플 먼저)
1. 이 taxonomy 결재 ← **현재 단계**
2. 20건 샘플 매핑 → 검색 화면에 꽂히는지 확인 → 미세 수정
3. 914건 전체 적용 (Code/검수)
4. mine-batch 동일 분류 생성
5. ai_draft 승인 화면 동일 분류 검수

## [부록] 현 914건 category → 5계열 매핑 분석 (Code, 2026-06-18 실측 기반)

현 `knowledge_entries.category` 22개 값(전부 approved, 합 914)을 5계열에 시범 매핑:

| 새 계열 | 매핑한 현 category | 건수 |
|---|---|---|
| 상품 | 상품(14)·상품구조특약(33) | 47 |
| 인수 | 인수(83)·인수심사(21)·계약청약고지(33) | 137 |
| 보장 | 보장(111)·보장담보(36)·손해사정보험금(43)·보상(8)·청구(7) | 205 |
| 비교 | (현 데이터 없음) | 0 |
| 상담 | (지식 category엔 없음 — 스크립트 소스) | 0 |
| **5계열 밖(미분류 후보)** | 계약(121)·판매(103)·민원(65)·세제(57)·세제과세(25)·재보험(32)·법령감독규정(25)·제도(11)·유지(8)·의무(6)·보험사(3)·(null 69) | **525** |

### 🚨 Code 판정 (결재 참고)
- **5계열이 현 914의 약 43%(389건)만 커버**, **나머지 57%(525건)는 5계열 밖**입니다(계약·판매·민원·세제·재보험·법령 등 = 실무·제도 영역).
- **비교·상담 계열은 현 지식 데이터에 0건**(비교=미생성 / 상담=스크립트 소스 영역).
- → 결재 선택지:
  - **(가)** 5계열 = 선택적 "필터 렌즈"로 두고, 525건은 **미분류(태깅 없음) 허용** — 검색은 소스 탭으로 다 잡히고, 상품/인수/보장 필터는 해당분에만. (taxonomy 원칙 "미분류 보류 허용"과 정합)
  - **(나)** 계열을 **보강**(예: 실무·제도·세제·민원 계열 추가) — 현 지식 절반 이상을 필터로 잡으려면 필요.
- 추천 = **(가)로 출발**(지금 과설계 X) → 20건 샘플 + 사용 패턴 보고 (나)로 보강 판단.

### 교차태그(축3) 스키마 — 결정 필요
질병·회사·상품군 교차태그는 현 스키마에 전용 컬럼이 없음(category 단일 컬럼만 확인). 다중 태그 저장하려면:
- 신규 컬럼(`tags text[]`) 또는 별도 태그 테이블 → **DDL = 대표님 결재 + Dashboard 직접 실행**.
- 20건 샘플이 "교차태그가 실제로 얼마나 필요한지"를 판정 → 그 후 스키마 결정(지금 미정).

### 다음 = 20건 샘플 매핑
대표가 결재하면, 아래 읽기 전용 SELECT로 카테고리별 대표 20건을 뽑아 6좌표로 손매핑 → 검색 화면 정합 확인:
```sql
SELECT entry_id, category, left(title,40) title, left(body,80) body_head
FROM knowledge_entries
WHERE status='approved'
ORDER BY category, entry_id
LIMIT 20;
```
(샘플은 category 다양성 위해 계열 대표값 위주로 추후 조정.)
