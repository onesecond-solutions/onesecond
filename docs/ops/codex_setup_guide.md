## 왜 그렇게 설정하는지 (총괄팀장 설명 — 대표님은 안 읽으셔도 됩니다)

### Setup script를 비워도 되는 이유
이 저장소는 **빌드가 없는 정적 사이트**입니다(GitHub Pages). `app.html` + `js/*.js` + `css/*.css`를 그대로 서빙합니다. 제품 코드 작업에 컴파일·번들 단계가 없어서 설치할 게 사실상 없습니다.
`package.json`은 보험이슈 RSS 페처(`scripts/issues/`) 전용이라 제품 코드 작업과 무관합니다. 코덱스가 그 스크립트를 건드릴 일이 생기면 그때 `npm install`을 넣으면 됩니다.
문법 검증은 `node --check js/파일.js`로 충분하고 node는 기본 제공됩니다.

### 🚨 Secrets(Supabase 키)를 넣지 않는 이유 — 중요
**일반 구현 작업에는 DB 키가 필요 없습니다.** 코덱스는 화면·기능 코드를 만들 뿐 DB를 건드리지 않습니다.
**키가 없으면 사고가 원천 차단됩니다** — 운영 데이터 오염·구버전 DB 접근·키 유출이 구조적으로 불가능해집니다.
DB 변경이 필요한 작업이 생기면 코덱스가 하지 않고 **총괄팀장에게 보고**합니다. 운영 DB는 기존 CI 채널(`.github/workflows/db-migrate.yml`) + 대표님 승인으로만 바뀝니다(이 절차도 그대로 유지).
→ 코덱스가 키를 요구하면 **범위를 벗어난 것**이니 멈추고 알려주세요.

### AGENTS.md는 대표님이 하실 일 없음
repo 루트 `AGENTS.md`에 코덱스 행동 규칙(역할·브랜치·배포·게이트·제품 코드 규칙·금지)이 박혀 있고, 코덱스가 매 작업마다 자동으로 읽습니다. 2026-07-17에 **일반 구현용으로 개정**했습니다.

## 운영 흐름

```
총괄팀장: 사양 확정 → 작업지시서 작성
   ↓
대표님: 지시서를 코덱스에 붙여넣기 (또는 총괄팀장이 전달)
   ↓
코덱스: feature 브랜치 작업 → 커밋 → PR 제출 (머지 안 함)
   ↓
총괄팀장: PR 검수 (범위·회귀·게이트·증거)
   ↓
대표님: 승인
   ↓
AI팀: 머지 → 배포 → 라이브 확인 → 완료 보고
```

- **코덱스는 PR까지.** 머지·배포는 하지 않습니다.
- **자동 머지 금지.** 대표님 승인이 게이트입니다.

## 파일럿 판단 기준

이 파일럿이 성공인지 볼 것:
1. **지시서 범위를 지키는가** (범위 밖 파일·기능을 건드리지 않는가)
2. **브랜치·PR 규칙을 지키는가** (main 직접 push 0)
3. **게이트를 지키는가** (새 화면이 게이트 밖으로 새지 않는가)
4. **증거 있는 보고를 하는가** (검증 없이 "완료"라 하지 않는가)
5. **총괄 검수에서 잡히는 회귀가 얼마나 되는가**

→ 몇 건 돌려보고 위 5개로 판정합니다. 안 맞으면 되돌리기 쉽습니다(설정 몇 개 + 문서).
