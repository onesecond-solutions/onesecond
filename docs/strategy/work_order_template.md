# Claude Code 작업지시서 표준 템플릿

> **목적:** Claude AI(총괄 기획자)가 Claude Code(실행 개발자)에게 발행하는 작업지시서의 표준 구조. 모든 작업지시서는 0번 항목(큰 그림 정합성 검증)을 첫 항목으로 반드시 포함한다.
>
> **배경:** 2026-04-28 사례 — AI가 메모리만 보고 추정으로 작업지시서 발행 → design_test 시안 승격 트랙(진짜 큰 그림) 놓침 → 6시간의 sweep + admin 작업 일부 무효화. 본 템플릿의 0번 항목은 이 사례 재발 방지 장치.

---

## 0. 큰 그림 정합성 검증 (모든 작업지시서 첫 항목 필수)

작업 시작 전 다음 검증 후 보고:

1. **`docs/sessions/_INDEX.md` 읽기** — 현재 큰 그림 한눈에 파악
2. 진행 중인 **메인 트랙·미해결 이슈·결정 대기 항목** 식별
3. 이 작업이 메인 트랙과 **정합인지 평가**:
   - 정합 (같은 방향)?
   - 별개 (다른 트랙, 충돌 없음)?
   - 충돌 (메인 트랙 진행 시 이 작업이 무효화될 가능성)?
4. **불일치 / 충돌 가능성 발견 시 즉시 멈추고 팀장님께 확인 요청**:
   > "_INDEX.md에 따르면 [X 트랙]이 진행 중인데, 오늘 요청하신 [Y 작업]은 [정합/별개/충돌] 관계로 보입니다. 의도가 맞는지 / 충돌 처리 방향을 확인 부탁드립니다."
5. 정합 확인되면 다음 항목으로 진행

⚠️ **정합성 검증 없이 작업 시작 금지.**
⚠️ "작은 작업이라 검증 생략" 금지 — 작은 작업도 큰 그림과 충돌 가능.

## 1. 작업 내용

(작업의 구체적 명세. 변경 대상 파일·영역·동작·UI 등 명확히 기술.)

---

## 2. 사전 환경 확인 (작업 시작 전 보고)

표준 항목:

1. 현재 브랜치가 `main`인지 확인
2. 워킹트리 깨끗 여부 (`git status`)
3. `main = origin/main` 동기 여부
4. 작업용 신규 브랜치 생성 (예: `feat/<scope>-<purpose>` 또는 `chore/<scope>-<purpose>` 또는 `sweep/<scope>-cleanup`)
5. (필요 시) 사전 매칭 / 영향 범위 카운트
6. (필요 시) 의존 토큰·매크로·외부 시스템 정의 확인

위 항목 보고 후 진행 승인 받고 다음 단계 진행.

---

## 3. 검증 단계

다음 검증 항목을 작업 후 실행:

- **A) 정량 검증**: 변경 의도가 수치로 검증 가능한 항목 (예: grep 0건, 파일 변경 라인 수)
- **B) 회귀 검수**: Claude Code 정적 분석 가능 영역 + 라이브 검수 필요 영역 분리 보고
- **C) 영향 범위 검사**: 변경 외 영역에 부수 영향 0건 확인
- **D) 매크로/외부 의존성**: (해당 시) 정상 작동 여부

⚠️ Claude Code 환경에서 라이브 검수 어려운 항목은 "검수 미수행" 명시.

---

## 4. 커밋 메시지

검증 완료 후 커밋. Conventional Commits 스타일 권장:

```
<type>(<scope>): <짧은 요약>

- 변경 항목 1
- 변경 항목 2
- 회귀 검수 결과

Refs: <근거 문서 경로>
```

타입 예시: `feat`, `fix`, `chore`, `refactor`, `docs`, `style`, `test`

---

## 5. 머지·푸시·정리

표준 chain:

1. `main` 체크아웃
2. `main` `--no-ff` <작업 브랜치> 머지 (슬롯 단위 추적용)
3. `origin main` 푸시
4. 작업 브랜치 로컬 삭제 (`git branch -d`)
5. (원격 푸시한 적 있으면) 원격 브랜치 삭제

---

## 6. 검토 후 보고 항목

표준 보고 항목:

- 사전 환경 확인 결과
- 사전 매칭 / 영향 범위 (작업지시서와 실측 비교)
- 정량 검증 결과
- 회귀 검수 결과 (각 액션별)
- 부수 영향 / standalone hex / 매크로 잔존 여부
- 머지·푸시·브랜치 정리 결과
- 다음 단계 옵션 (있으면)
