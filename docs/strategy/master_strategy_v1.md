## §1. 한 줄 정의

### 1줄
> onesecond = **보험업 운영 흐름 네트워크 + 보험 검색 인프라 + 반복 질문 감소 시스템**

### 5줄
- onesecond는 단순 보험 게시판·CRM·스크립트 SaaS가 아니다
- **현장 질문 흐름 + 보험사 공지 흐름 + 인수/상품 정보 흐름 + 반복 질문/답변 구조** 를 하나의 운영 네트워크로 연결한다
- 정보를 모으는 것이 아니라 **검색 가능하게 구조화** 하는 것이 본질
- 사용자(설계사)의 상담 흐름을 멈추지 않게 하고, 보험사 임직원의 반복 응대 피로를 줄인다
- 양쪽 가치가 한 시스템에서 동시 발생 = **진입장벽**

### 15줄 (시스템 본질 + 영업 본질 통합)
- 17년 보험 현장(대면+TM)에서 출발한 도메인 통찰: **상담은 멈추는 순간 깨진다**
- 그래서 onesecond = **상담 흐름 OS** (시스템 측면, OS 정의 v2)
- 동시에 **반복 질문 감소 시스템** (영업 측면, 본 문서)
- 사용자(설계사)는 4 레이어 중 3개만 본다: 실장님 공지 / 네비방 / 스마트 게시판
- **보험사 게시판은 인큐베이션 단계에서 admin 외 비공개**
- 실장님이 카테고리(공지/인수/상품) 선택해 글 작성 → **일방향 자동 저장** → 스마트 게시판 노출
- 네비방 질문 → 보험사 게시판 자동 전달 → 보험사 답변 → 네비방 자동 반영 (역방향 흐름)
- 스마트 게시판 = 일반 사용자 글 작성 X. 오직 admin이 네비방에서 정제·이전
- 이 흐름이 6개월~1년 자연 가동되면 → 보험사 매니저·지점장이 자발 발견 → 자발 입점
- 영업 화법: "매일 같은 질문, 같은 답변 힘들지 않으세요?"
- 가격 = **게시판 사용료 X** / **운영 효율 + 검색 인프라 비용 O**
- 3티어: 100만원(채널) + 100만원(소식지 검색화) + 100만원(자료 통합 검색화) — 시장 검증 전 수치
- 31사 동시 단계 진입 정책 + 사용자 화면 편향 노출 방어 = 0순위 정체성 보호
- 모바일 퍼스트 + 카톡보다 강력한 푸시 + C영역 미열람 공지 피드 = 일상 알림과 분리된 **업무 집중 알림 센터**
- 4팀(AZ 더원) = 5/15 첫 사용자 검증 + 17년 도메인 + admin 본인 = **최적 측정 환경**

## §2. 0순위 브랜드 정체성 (모든 결정의 기준)

### ✅ 중립 독립 SaaS

**onesecond는 어느 보험회사·GA(대리점)에도 소속되지 않은 독립 SaaS이며, 모든 사업 결정은 특정 회사 이해관계가 아닌 "현장 전체 기준"의 중립 위치에서 내린다.**

### 왜 독립성이 중요한가
- 보험 현장 = 다양한 보험사·GA 이해관계가 충돌하는 영역
- 특정 회사 종속 시 → 그 회사 영업 도구로 전락 → 다른 회사 설계사 거부
- onesecond = 모든 보험 현장이 사용 가능한 **운영 OS**

### 4팀 ↔ onesecond 관계 (오해 방지)
- 4팀(AZ 더원 GA의 한 팀, 약 40~50명) = 5/15 첫 사용자 검증
- 영구 단독 고객 X / AZ 종속 X / GA 편향 X
- 향후 다른 GA / 다른 보험사 매니저 입점 시 동일 OS

### 회귀 신호 (즉시 보고)
- ❌ 특정 보험사 기능 중심 사고 (NH 전용 / 삼성생명 편향 / 메리츠 단독)
- ❌ 특정 GA 편향 UX
- ❌ 폐쇄형 조직 SaaS 해석
- ❌ 4팀 = 영구 단독 고객 가정

## §3. 시스템 본질 5종 — 사용자 인지 vs 실제 내부 구조

### 사용자에게 보이는 구조
사용자(설계사)는 다음 3개만 인지:
1. **실장님 공지**
2. **네비게이션방**
3. **스마트 게시판**

→ **보험사 게시판의 존재 자체를 모른다.**

### 실제 내부 흐름

```
[실장님 공지 작성]
   ├─ 카테고리 선택 (공지 / 인수 / 상품)
   ↓
[보험사 게시판 해당 카테고리로 일방향 자동 저장]  ← admin 외 비공개
   ↓
[스마트 게시판로 자동 노출]
   ↓
사용자 확인

────────────────────────────────────────

[네비게이션방 사용자 질문]
   ↓
[해당 보험사 게시판으로 자동 전달]
   ↓
[(향후) 보험사 임직원 답변]
   ↓
[네비게이션방으로 자동 반영]
   ↓
사용자 확인
```

### 핵심 정정: ❌ 양방향 미러링 → ✅ 일방향 자동 저장
- 미러링(양쪽 동기화) X — 팀장님 본인이 작성·삭제 권한 보유 위해
- 일방향 자동 저장 = 보험사 게시판 = **저장소**, 스마트 게시판 = **노출**
- 이 단순화로 데이터 무결성 단일 소스 + 작업 흐름 명확

### 5종 본질 통합 표

| # | 본질 | 회귀 금지 |
|---|---|---|
| 1 | 스마트 게시판 = 정제된 실전 데이터 (admin write-only) | 커뮤니티형 게시판 X / CRM X |
| 2 | 네비게이션방 = 질문 생성 엔진 + 실시간 컨트롤타워 | 단순 채팅방 X |
| 3 | 보험사 게시판 = 공급 레이어 (인큐베이션 단계 비공개) | 메인 서비스 X / 사용자 직접 진입 X |
| 4 | 디지타이징 = 검색 가능 구조화 = 진입장벽 | 단순 PDF 보관 X |
| 5 | 모바일 퍼스트 + 강력한 푸시 + C영역 미열람 피드 | 데스크탑 퍼스트 회귀 X |

## §4. 4 레이어 역할 분리

### A. 실장님 공지
**역할:** 운영 방향 레이어

**목적:**
- 오늘의 방향
- 긴급 체크
- 팀 운영
- 현장 공유

**특징:** 보험사 상품/인수 공지를 단순 반복 전달 X. 보험사가 공급한 정보를 **현장 흐름 + 오늘의 방향 + 실제 행동**으로 연결.

> 정보 전달 < **운영 방향 제시**

### B. 네비게이션방
**역할:** 질문 생성 엔진

**목적:** 사용자가 상품·인수·전산·민원·녹취·운영 등 실제 현장 질문을 가장 빠르게 생성하는 공간

**특징:**
- 카테고리보다 **"상황 선택"**
- 질문 흐름 유지
- 단톡방 현실 UX 디지털화

### C. 스마트 게시판
**역할:** 정제된 실전 데이터 레이어

**특징:**
- 일반 사용자 글 작성 **불가** (RLS 강제)
- 오직 admin만 선별 등록
- 네비게이션방에서 엄선된 질문만 이동

> 원본 흐름 = 네비게이션방 / 정제 데이터 = 스마트 게시판

### D. 보험사 게시판 (내부 공급 레이어)
**역할:** 보험사 공급 레이어 (인큐베이션 단계 = admin 외 비공개)

**특징:**
- 초기 사용자 비공개
- 보험사 직원도 존재 자체를 모를 수 있음
- 실장님 공지/자료가 자동 저장되는 내부 구조

**향후 목적:** 보험사 임직원이 직접 공지·답변·상품/인수 정보 공급하는 구조의 기반

## §5. 인큐베이션 전략 — "처음부터 공개하지 않는다"

### 왜 처음부터 공개 X
- "보험사 공식 게시판 참여해주세요" 접근 시:
  - 부담 / 승인 문제 / 정치적 이슈 / 공식 운영 여부 / 책임 문제 발생
- 강제 영업 = 종속성 + 영업 비용 폭증

### 5단계 자발 입점 흐름

```
1단계  실장님이 먼저 운영
        ↓
2단계  질문/공지 흐름 생성 (네비방 + 스마트 게시판 가동)
        ↓
3단계  스마트 게시판 데이터 축적 (검색 가능)
        ↓
4단계  보험사 실무자가 자연스럽게 관심
        ↓
5단계  "사실 이런 시스템이 이미 돌아가고 있습니다" 시연
```

### 핵심
> 말로 설득하지 않는다. **이미 굴러가는 흐름을 보여준다.**

## §6. 보험사 영업 시나리오

### 영업 화법 (17년 현장 경력에서 나온 정확한 pain point)

> **"매일 같은 질문, 같은 답변 힘들지 않으세요?"**

### 보험사 실무자 실제 pain point
- 반복 질문 응대
- 카톡 피로
- 자료 재전송
- 과거 답변 검색 불가
- 신입 매니저 학습 곡선

### onesecond 가치
사용자뿐 아니라 **보험사 실무자의 피로도**도 함께 줄여주는 구조 = 양면 가치

### 5단계 체험 흐름 (락인 검증 지표)

```
1단계  게시판처럼 시작
        ↓
2단계  "검색되네?"        ← 디지타이징 가치 첫 체감
        ↓
3단계  "예전 자료 바로 찾네?"  ← 검색 인프라 가치 체감
        ↓
4단계  "반복 질문 줄었네?"   ← 운영 효율 가치 체감
        ↓
5단계  "이거 없으면 불편한데?" ← 락인 완성
```

### 영업 시나리오 = 가격 인상 타이밍 = 락인 검증 지표
이 5단계는 단순 marketing copy X. **언제 가격 인상 가능한가**의 검증 지표:
- 1단계 → 2단계 = 1번 결제 (월 100만원)
- 2단계 → 3단계 = 2번 결제 (+ 100만원)
- 3단계 → 4단계 = 3번 결제 (+ 100만원)
- 5단계 도달 = **재계약 협상력 = onesecond 측 우위**

## §7. 3축 수익 모델

| 축 | 대상 | 가격 | 본질 |
|---|---|---|---|
| **1축** | 설계사 (개인) | pro 9,900원/월 | 상담 흐름 + 스마트 설계서 |
| **2축** | 보험사 (광고주) | 미정 | 배너 광고 (보험사 게시판 인큐베이션과 충돌 가능 — 결재 대기) |
| **3축** | 보험사 (월 플랫폼 비) | 100~300만원/월 (시장 검증 전) | 운영 효율 + 검색 인프라 |

### 축별 정합
- 1축 = pro 분리 = 무료 onesecond의 정체성 보호 장치
- 2축 = 인큐베이션 단계에서는 가동 X (보험사 게시판 비공개와 정합 안 됨) — 5단계 자발 입점 후 결재
- 3축 = 본 문서 § 8 상세

### 31사 × 3축 동시 검증 정책
- 어느 한 보험사가 어느 한 축으로 결제해도, **사용자 화면 편향 노출 0** 강제
- 결제 보험사 라벨 노출 X (예: "메리츠 [PRO]" 표기 금지)
- 이 정책 = 0순위 정체성 보호 = 절대값

## §8. 3티어 가격 시나리오 v0

> ⚠️ **본 가격 = 시장 검증 전 "내 바람" 수치.** 4팀 오픈 후 6개월~1년 파일럿(2~3개 보험사) 후 정식 가격 책정.

### 1단계 — 게시판 사용 권한
**100만원/월 (예상)**

포함:
- 게시판 사용
- 소식지 업로드
- 공지/인수/상품 자료 게시

→ 보험사 운영 채널 역할

### 2단계 — 소식지 검색 최적화
**+ 100만원/월 (예상)**

포함:
- PDF 단순 업로드 X

## §10. 모바일 퍼스트 + 푸시 + C영역

### 핵심 철학
- **카톡 = 일상 + 업무 섞임** → 보험 알림톡도 그냥 지나침
- **onesecond = 업무 집중** → 푸시 알림 한 번 더 확인하게 됨
- → **업무 알림 집중도 = onesecond > 카톡**

### 핵심 전략 4건
1. **모바일 퍼스트** (디자인 무너져도 OK)
2. **강력한 푸시 알림** (Web Push API + FCM 등 검토)
3. **즉시 확인 구조** (모바일에서 본 내용 → PC 검색으로 다시 확인 가능)
4. **C영역 = 업무 알림 센터**

### C영역 활용 (현재 사이트 빠른 실행 버튼 아래 공간)

로그인 시 C영역에 노출:
- 미열람 공지
- 보험사 변경
- 실장님 공지
- 내 질문 답변
- 긴급 공지

→ **업무 알림 센터** 역할 = 푸시 못 본 내용 PC에서 줄줄이 확인

### 구글 연동 본질
- 카톡 = 일상 섞임 → 푸시 무력화
- 구글 = 업무 계정 분리 → 푸시 집중도 ↑
- 구글 OAuth = 단순 로그인 편의 X / **업무 채널 집중 전략**

## §15. 정체성 — "딸깍 = one + second"

### 본진 (2026-05-14 팀장님 본인 명시 박힘)

> "딸깍은 그냥 하는 소리가 아니야. 원세컨드 = 딸깍으로 만들 거야."
>
> "원세컨드 네이밍도 중요하지만 사람들이 쉽게 말할 수 있는 네이밍이 필요했는데 딸깍이 제일 좋은거 같애."

원세컨드 정체성 = **딸깍**. one + second 본진. 단순 의성어 X — **브랜드 본질 + UX 본진 + 슬로건 + 일상 호칭**.

### 두 자리 네이밍 본진 (역할 분담)

| 자리 | 네이밍 | 본진 |
|---|---|---|
| **정식 브랜드** | 원세컨드 (onesecond) | 공식 SaaS / 도메인(onesecond.solutions) / 계약 / 정식 문서 |
| **일상 호칭** | 딸깍 | 사람들이 쉽게 말함 / 마케팅 슬로건 / UI 본인 박힌 자리 / 입소문 본진 |

**Why 두 자리 박힌 자리:**
- 원세컨드 = 발음 박힌 자리 길어 일상 박힘 격차 박힘
- 딸깍 = 2음절 + 의성어 본진 = 사람들이 본인 박지 X 박힘 + 입소문 전파 정합
- 본 자리 = 인스타·당근 같은 줄임 호칭 본진 정합 ("당근" = 당근마켓)

### Why (왜 본진인지)

본 자리 = 사용자 본질 정합 박힘 자리. 보험 영업 운영의 모든 작업이 "딸깍 한 번"으로 끝나야 한다.

- **통화 중** = 한 손 박힘 + 1초 안 박힘 정합
- **단톡방 흐름 중** = 흐름 박지 X 박힘 정합
- **영업 자리 사이** = 길게 박는 클릭 박지 X 박힘 정합

딸깍 한 번 = 사용자 흐름 박지 X 정합. 다단계 박힌 자리 = 정체성 회귀.

### How to apply (매 UI 결정 시 점검)

- **모든 UI 결정 시** = "본 자리 딸깍 한 번으로 끝나는가?" 점검 본진
- **입력 모달** = 딸깍 → 본진 박힘 → 딸깍 → 박음 (3 step 박지 X 본진 정합)
- **이동** = 딸깍 한 번 정합
- **검색** = 입력 + 결과 = 딸깍 흐름 안 박힘 정합

### 학습·중독 본진 (2026-05-14 본인 명시 박힘)

> "신규 사용자들이 딸깍만 보고 클릭해보면 자동으로 학습되게끔 반복 패턴 학습 및 중독시키는거야"

본 자리 = §15 본진 강화 박힘. 사용자 onboarding 본진 정합:

- **반복 패턴 박힘** = 모든 자산(메시지/사진/동영상/파일/링크/일정 등) 동일 [딸깍] 버튼 박힘 → 사용자 본진 "어디든 딸깍 = 동일 동작" 학습 박힘
- **자기 발견 학습** = 신규 사용자 = "딸깍" 텍스트만 보고 클릭 → 액션 메뉴 박힘 → 본인 학습 박힘 (튜토리얼 박지 X 정합)
- **중독 회로** = 1초 안 박힘(딸깍) + 즉시 피드백(오버레이) + 일관 패턴(5자산 동일) = 도파민 회로 강화

**매 UI 결정 시 추가 점검:**
1. 신규 사용자 본인 박힘 가능한가? (튜토리얼 박지 X 박혀 본인 학습 박힘 정합)
2. 동일 패턴이 다른 자리에도 박혀 있는가? (반복 패턴 박힘 = 학습 강화)
3. 즉시 피드백 박혀 있는가? (1초 안 박힘 + 시각 본진)
4. 본진 박힌 자리 = 사용자 본진 박힘? (어디 박힐지 직관 박힘)

### 정체성 회귀 신호 (§14 정합)

본 본진 정합 박지 X 박힌 자리 박힐 시 즉시 보고:
1. 다단계 입력 본진 박힘 (3 step+)
2. 클릭 → 모달 → 확인 → 박음 (4 step+)
3. 본 자리 박지 X 박혀 사용자가 한 번에 박지 X 흐름
4. "두 번 클릭 본진" 박힘 자리 trade-off 사전 결재 박지 X
5. **자산마다 다른 액션 박힘** (반복 패턴 깨짐 = 학습 격차)
6. **튜토리얼 박힘 자리 박혀야만 본진 박힘** (자기 발견 박지 X)
7. **즉시 피드백 박지 X** (도파민 회로 격차)

### 기존 본진과의 관계

- **§10 모바일 퍼스트** — 한 손 박힘 정합 (딸깍 한 번 본진 강화)
- **§11 진짜 권력** — "보험 종사자가 1초 안 박힘 자리" 본진 = 본 §15 본진 정합
- **§13 결재** — 본 정체성 본진 정합 박지 X 박을 시 결재 박을 자리
