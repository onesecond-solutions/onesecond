# PART A — 진단

## A-1. 화면 점유율 비교 (모바일 캡처 기반, 2026-05-14)

**입력:** 임태성 팀장님 모바일 화면 캡처 2장 (카톡 더원 상품방 + onesecond.solutions)

| 영역 | 카톡 | onesecond 현재 |
|---|---|---|
| 채팅 본문 | **~88%** ✅ | ~40% ⚠️ |
| 상단 부가 (핀·공지) | ~6% | **~28%** 🚨 |
| 입력창 | ~6% | ~14% |
| 하단 메뉴 | 0% (햄버거) | ~12% |

**진단:** 부가 UI가 40%를 먹어 채팅(본질)이 절반도 안 됨.

## A-2. 핵심 격차 (캡처에서 즉시 보임)

1. **핀 공지 박스 28% 점유** — 의도는 좋으나 항상 펼친 상태로 공간 폭락
2. **하단 메뉴(B영역) 항상 노출** — 12% 고정 차지
3. **입력창 안내 텍스트 3줄** — "Enter 전송 / Shift+Enter 줄바꿈" 항시 노출
4. **메시지 버블 좌우 여백 과다** — 한 줄당 글자수 적어 "조회때 말씀드린 일배책 사 / 례들" 식 끊김

# PART B — 핵심 통찰

## B-1. 사용 맥락 결정적 차이 (모든 전략의 출발점)

| 차원 | 카톡 | onesecond |
|---|---|---|
| **목적** | 사적 대화 | 업무 실행 |
| **양손** | 자유 | 한 손 (전화기 든 손은 귀에) |
| **시선** | 화면 집중 | 화면 + 고객 말 동시 처리 |
| **시간** | 느긋함 | 3초 안에 답 찾기 |
| **입력** | 자주 | 거의 안 함 (읽기만) |
| **메시지 운명** | 휘발 | 자산 축적 |
| **사용자가 찾는 것** | 지금 말 거는 사람 | 답을 빨리 찾아 통화 복귀 |

## B-2. 결론

> **카톡 ≠ onesecond. 채팅 UI를 빌릴 뿐, 카톡이 되려 하지 마라.**
> **onesecond는 '읽기 위한 채팅'이고, 메시지 하나하나가 자산이다.**

카톡 디자인 팀이 onesecond를 본다면 첫 마디:
> "이건 메신저가 아니라 상담 운영 OS네요. 카톡 흉내 내면 망합니다."

# PART C — 5대 전략 축 (onesecond 차별화)

## 축 1. 카톡은 "사람 중심", onesecond는 "맥락 중심" ⭐⭐⭐

### 처방
메시지 상단에 **맥락 태그** 노출.

```
[🔴 긴급] [원수사·메리츠] [실손 분쟁]
        570412 남자 / 직업1급 / 고혈압
        한쪽 절단 18일입원 ...
        ─────────────────
        └ 답변 3건 / 채택 1건 ⭐
```

### 효과
- 통화 중 폰을 살짝 봐도 맥락 즉시 인지
- 6필드 단계형 입력 (OS v2 §7 정합) + 보험사 공급 레이어 (§10) 정합
- 카톡엔 절대 없는 패턴 — 구조화 메시지라야 가능

### Code 검증 필요
- 메인 트랙 2 게시판 본진 Step 7~9에 맥락 태그가 spec에 포함되었는지
- board_type 7종 (manager_notice / field_voice / insurer / ...) 과 매핑 가능 여부

---

## 축 2. 카톡은 "최신 메시지", onesecond는 "채택된 답변" ⭐⭐⭐

### 처방
⭐ 채택 답변 상단 고정 + 가로 캐러셀.

```
┌─ ⭐ 채택 답변 (3건) ──────────┐
│ ▶ 메리츠 표준체 검토 가능     │  ← 좌우 스와이프
│   ● ○ ○ (1/3)                │
└────────────────────────────────┘
─────────────────────────────────
[일반 메시지 시간순]
```

### 효과
- OS v2 §7 "질문이 구조적으로 축적되는 것" 정합
- OS v2 §9 "정리된 운영 자산 저장소" 시각화
- 카톡엔 "정답" 개념 자체 없음 → onesecond 본질 차별화

### ⚠️ Code 검증 필수
- **"채택 답변" 시스템이 현 spec에 명문화되었는지** — 미확인
- 없으면:
  - 옵션 (a) 메인 트랙 2에 신규 추가 (분량 영향 검토)
  - 옵션 (b) 별 트랙으로 분리 (5/18 후 후속)

---

## 축 3. 카톡은 "스크롤", onesecond는 "필터" ⭐⭐

### 처방
핀 공지 자리를 **필터 칩**으로 재활용.

```
┌──────────────────────────────────┐
│ 🔍 검색                          │
│ [전체] [⭐채택] [내질문] [원수사] │ ← 필터 칩
│ [메리츠] [DB] [한화] [삼성생명]  │ ← 보험사 분기
└──────────────────────────────────┘
```

### 효과
- **핀 공지 28% 점유 → 필터 칩 12%로 압축** (16% 회복)
- "메리츠 사례만" 탭 = 즉시 슬라이싱
- onesecond만 가능한 이유: 6필드 구조화 데이터라야 칩 분류 가능

### Code 검증 필요
- 현재 핀 공지(최대 10건) 시스템과 필터 칩이 양립 가능한지
- insurer_list 단일 source 정합

---

## 축 4. 카톡은 "양손 여유", onesecond는 "한 손 통화 중" ⭐⭐⭐

### 처방 3종

**a. 좌측 가장자리 스와이프 → 메뉴 호출**
- 평상시 메뉴(B영역) 완전히 숨김
- 엄지로 좌측 가장자리 우측 스와이프 = 한 손 조작

**b. "통화 중 모드" 토글**
```
[일반 모드]               [통화 중 모드 ON]
채팅 70%                  채팅 95%
입력창 15%                입력창 사라짐
메뉴 15%                  메뉴 미니 아이콘만
```

**c. 폰트 +20% 자동 + 음성 알림**
- 통화 중 = 폰 멀리서 봄 → 폰트 확대
- 새 답변 도착 시 진동 + 짧은 음성 ("답변 도착")

### 효과
- 카톡엔 절대 없는 본질 차별화
- OS v2 §1 "상담 중 멈추지 않게 하는 것" 정합

### ⚠️ Code 검증 필수
- B영역 정의 = "좌측 사이드바" — 모바일에선 어떻게 변환되어 있나
- 변환 규칙이 spec에 명문화되었는지
- 통화 감지 API (Web API / PWA 한계)

---

## 축 5. 카톡은 "휘발", onesecond는 "자산" ⭐⭐⭐

### 처방
메시지 길게 누르기 = 4단 자산 전환.

```
┌──────────────┐
│ ⭐ 채택 답변   │  → 게시판 상단 고정
│ 📋 스크립트로  │  → MY SPACE 스크립트 저장
│ 🔖 북마크     │  → 개인 즐겨찾기 (무제한)
│ 📤 고객 공유   │  → PRO 스마트 설계서 전환
└──────────────┘
```

### 효과
- OS v2 §17 "질문 → 검색 → 스크립트 → 설계서 → CRM" 흐름이 액션 하나로 완결
- 핀 등록 10건 제한 → 북마크 무제한
- 앱 내 자산 전환 완결 (카톡은 외부 앱 필요)
- 4팀 자산화 트랙(카톡 1,564건 마이그레이션) 정합

### Code 검증 필요
- personal_memos 본진과 북마크 시스템 정합
- PRO 스마트 설계서 전환은 free 정체성 보호 정책 위반 X 확인

5대 전략 축 가동하려면 채팅 40% → 75% 회복 필수.

| # | 처방 | 회복 면적 | 난이도 | 5/18 가능 |
|---|---|---|---|---|
| 1 | 핀 공지 → 아이콘 + 바텀시트 (또는 축 3 필터 칩으로 재활용) | +23% | 중 | ✅ |

| 4 | 메시지 버블 좌우 여백 축소 + 시간 라벨 압축 | +6% | 하 | ✅ |

**누적 효과:** 채팅 40% → **75%** (카톡 88%에 근접)

# PART E — 추가 논의 2항목 (2026-05-14 PC 진입 시점)

## E-1. 채팅창 입력폼 — 매니저 공지 입력폼 재사용

### 팀장님 의견
> "채팅창 입력폼은 기존 매니저 공지에서 사용했던 입력폼 재사용"

### 인지 격차
- 모바일 세션 Claude AI는 매니저 공지 입력폼 구체 구현 직접 확인 X
- Code가 실파일 확인 후 결정 필요

### Code에게 의뢰할 확인 항목

```
1. 매니저 공지 입력폼 위치 (파일·라인)
2. 현재 입력폼 구조 (단일 줄/멀티 줄, 첨부 가능 여부, 전송 버튼 위치)
3. 권한 분기 로직 (누가 쓸 수 있나)
4. 모바일 반응형 동작 (768px 이하 어떻게 보이나)
5. 토큰·CSS 클래스 정의
```

### 재사용 전략 3종

| 전략 | 내용 | 장점 | 단점 |
|---|---|---|---|
| **A. 통째 재사용** | 매니저 공지 입력폼 컴포넌트 그대로 박음 | 일관성 + 작업량 최소 | 공지용 UI가 빈번한 채팅에 맞을지 검증 필요 |
| **B. 골격 재사용 + 모바일 압축** | HTML·이벤트는 재사용 + 768px 이하 공간 압축 4선 적용 | 일관성 + 모바일 최적화 양립 | 분기 처리 필요 |
| **C. 컴포넌트화 + 모드별 호출** | `mode="notice"` / `mode="chat"` / `mode="mobile-chat"` | 향후 확장(스크립트·게시판·CRM) 모두 재사용 | 초기 추상화 비용 |

### 권장
- **5/18 시급 + 본 PC 영구 진실 정합 → 전략 B**
- 전략 C는 5/18 후 별 트랙

### 단톡방 의뢰서 시나리오 결재 연계

- 시나리오 A/B/C 결재 시 입력폼 전략도 동시 결정

## E-2. 푸시 알림 기능 강화

### 팀장님 의견
> "푸시 알람 기능 강화"

### 인지 (메모리·세션 패키지 기반)
- 인증 메일 = Resend Custom SMTP (5/15 → 5/18 shift 추정)
- Sentry 에러 모니터링 (별 트랙 #22)
- **푸시 알림(브라우저/모바일) 자체는 spec 명문화 X로 추정** — Code 확인 필요

### 5단계 설계

#### 단계 1. 기본 푸시 (모든 SaaS 공통)
- 새 메시지 도착 / 채택 답변 등록 / 매니저 공지 등록
- → 누구나 한다. 차별화 0

#### 단계 2. 맥락 기반 알림 ⭐ onesecond 차별화 1단

**카톡 알림:** `"[더원 상품방] 미래에셋_최세화: 암담보만 표준체 검토..."`
→ 통화 중 봐도 무슨 종류 답변인지 즉시 인지 불가

**onesecond 차별화:**
```
🔴 긴급 답변 도착
[원수사·메리츠] [실손 분쟁]
570412 남자 고혈압 → 표준체 가능
👤 미래에셋_최세화 · 5초 전
```

**기술:** Web Push API (FCM/VAPID) + Service Worker `notification.show()` 커스텀 페이로드

#### 단계 3. 우선순위 알림 ⭐⭐ onesecond 차별화 2단

**문제:** 4팀 215건/일 메시지 중 알림 받아야 할 건 5~10건

| 등급 | 트리거 | 알림 형태 | 사운드 |
|---|---|---|---|
| 🔴 긴급 | 내 질문에 채택 답변 등록 | 진동 + 강한 사운드 + 락스크린 | 별도 음 |
| 🟠 중요 | 매니저 공지 / 내 멘션 | 진동 + 일반 사운드 | 기본음 |
| 🟡 일반 | 일반 메시지 | 진동만 (무음) | 없음 |
| ⚪ 사일런트 | 보험뉴스 / 통계 | 배지만 | 없음 |

#### 단계 4. 야간 운영 모드 + 통화 중 모드 ⭐⭐⭐ onesecond 본질 차별화

**a. 야간 운영 모드 (스케줄 기반)**
- 22:00 ~ 익일 08:00 = 🔴 긴급만 + 🟠 중요는 묶음 푸시 (다음날 아침)
- 🟡 일반·⚪ 사일런트는 차단
- 가족 옆에서 화면 미리보기 차단 (제목만 표시)

**b. 통화 중 모드 (실시간)**
- 폰 통화 감지 시 자동 진입 (네이티브 PWA 한계 검토 필요)
- 진동만 / 화면 켜짐 X
- 통화 종료 후 30초 내 묶음 푸시 ("통화 중 도착 5건")

**c. 운영 시간 외 자동 응답 (PRO 트랙 가능성)**
- 매니저 부재 시 "내일 오전 답변" 자동 전송

#### 단계 5. 알림 → 액션 즉시 연결 ⭐⭐⭐ 카톡 절대 못하는 차별화

**카톡:** 알림 탭 → 앱 열림 → 채팅방 찾기 → 메시지 읽기 (4단계)

**onesecond:** 알림 자체에 **인라인 액션 버튼**

```
🔴 긴급 답변 도착
[메리츠] 표준체 검토 가능
👤 미래에셋_최세화

[⭐채택]  [📋스크립트로]  [💬빠른 회신]
```

**기술:** Web Push Notification Actions API

### 구현 난이도 + 5/18 가능 여부
