# [토론] 한 줄 요약형(glance-row) 패턴 — 사이트 전역 구현

- 경로: `docs/discussions/2026-06-26_glance-row-pattern.md`
- 참여: 기획팀장(Web) ↔ 총괄팀장(Code) / 대표님 결재
- 상태: **논의 중** (구현 착수 아님 — 스페이스 색상 다음 트랙). 기록 방식 = 한 파일에 번갈아 누적.

---

## A. 기획팀장 제안 (문제 + 방향)

**[문제]**
- "한 줄 요약(글랜스)"을 사이트 전역에 깔고 싶음.
- 현재: 카드는 만들 수 있으나, 클릭 시 보여줄 "상세"가 없음 (목적지 부재).
- 위험: 화면마다 상세를 따로 만들면 또 파편화 → "부품 1개 + 목적지 1개로 수렴" 원함.

**[방향 제안]**
1. 한 줄 요약을 공용 부품 하나로 규격화 (가칭 `glance-row`)
   구조: `[●출처색] [공간/출처명] [요약 or 카운트] [상세 →]`
   - 마이의 팀/지점 요약 · 검색 결과 · 공지 목록이 전부 같은 부품으로.
2. 클릭 목적지 = 새로 만들지 말고 기존 그릇 재사용 (검색 결과의 마스터-디테일 우측 패널 또는 바텀시트 중).
3. 상세 동작 = 혼합: (A) 미리보기=우측 패널에 상세만(기본) / (B) "전체보기"만 해당 스페이스로 점프.
4. 착수 시점: 스페이스 색상 머지·검수 끝난 다음 (동시 금지·회귀 위험).

## B. 총괄팀장에게 묻는 것 (코드 현실 — 추측 구현 전)

- Q1. 검색 결과 우측 패널(마스터-디테일)을 글랜스 클릭 목적지로 재사용 가능한가? 검색 전용인가, 다른 컨텍스트(마이·공지)에서도 호출 가능한가?
- Q2. 지금 "카드"를 만드는 컴포넌트/함수 이름·위치는? 한 줄(glance-row) 변형을 얹을 수 있나, 신설인가?
- Q3. 한 줄 카운트(공지 N·새 자료 N·답변대기 N)를 지금 쿼리로 뽑을 수 있나? (공지 카운트 guide 제외 규칙 포함)
- Q4. 부품이 들어갈 화면 수·전 스페이스 영향? 색상 작업과 겹치는 파일이 있나?

## C. 총괄팀장 응답 (2026-06-27, 코드 근거)

### A1 (Q1 우측 패널 재사용) — **부분 가능** (골격 OK, 컨텍스트 분기 일반화 필요)
- 상세 렌더 함수 `_searchOpen(table, id)`(app.html:10709)는 **table·id만 받아 동적 쿼리로 그리는 범용 골격** — 이 부분은 재사용 가능.
- 그러나 "어느 패널에 그릴지"가 검색에 묶임: `_svTarget==='hsResults'`(검색 홈 2단) 분기 + 전역상태 `_svPanelHtml`·`_svOpenKey` 의존. 우측 패널(`#svPanel`)·2단 레이아웃(`sv-2col`/`sv-left`/`sv-right`)은 **검색 홈(#v-home 검색중)에만 존재**.
- **결론:** "table/id 넘기면 상세 그린다"는 재사용 OK. 단 마이/공지에는 우측 패널 그릇 자체가 없어, 거기서 글랜스 클릭 목적지로 쓰려면 `_searchOpen`에 **target 인자**를 더해 "검색 2단 / 범용 컨테이너"를 분기해야 함(또는 바텀시트로 통일).

### A2 (Q2 부품) — **신설 권장** (`.glance-row` 신규)
- 가장 가까운 기존 부품 = 검색 결과 행 `.sores`(CSS 965, 생성 `_svRender` 10549): `[태그][제목][스니펫 2줄][메타]`. **스니펫 2줄 포함**이라 "정확히 1줄 + 카운트 + [상세→]"인 글랜스와 다름.
- 좌측 목록 `.item`도 2행(제목/메타). 스페이스 카드 `.sn-card`·현황 `.sn-dcard`는 세로 2단.
- **결론:** 1줄 전용 `.glance-row` **신규 클래스**가 깔끔(`[●색][명][카운트][상세→]`). 생성은 `_snMyspaceShow`(5413)/`_snScopeShow`(5478) 안 또는 공용 헬퍼.

### A3 (Q3 카운트) — **부분 가능** (범용 헬퍼 있음, 일부 쿼리 신규)
- 범용 카운트 헬퍼 `_diaCount(url)`(13255)이 이미 있음(`count=exact` 헤더). `loadDiaryDash`(13256)가 이걸로 공지·답변대기 카운트 → `data-dcnt` 미러.
- **단 현 공지 카운트(13261)는 `team_notices?deleted_at=is.null` = scope 필터 없음(전체)** → 팀/지점별은 `scope=eq.team/branch` 추가 필요.
- **guide 제외:** 스페이스 설명서(spguide)는 `team_notices` 테이블이 **아니라** 별도(`_spIntroRender`) → team_notices 카운트에 **애초에 안 들어감**. guide 필터 불필요(걱정 없음).
- **"새 자료 N":** `myspace_files?scope=eq.{scope}&created_at=gte.{기준}` **신규 쿼리 필요**(미구현). 답변대기 Q&A 카운트는 기존 그대로 사용 가능.
- **결론:** `_diaCount` 재사용 + scope/날짜 필터만 추가하면 카운트 산출 가능.

### A4 (Q4 영향) — **전부 app.html 단일 파일**
- 들어갈 화면: 마이/팀/지점 요약(`_snMyspaceShow`/`_snScopeShow`)·검색 결과(`_svRender`)·공지(`_spCospShow`) → **전 스페이스**.
- 색상 작업과 같은 app.html이나 **영역은 다름**(색상=토큰/`sn-page`/`sn-card` / 글랜스=신규 `.glance-row`+카운트 쿼리). 직접 충돌은 적으나 같은 파일이라 **순차 진행 권장**(색상 2단계 끝난 뒤 글랜스).

### 막히는 점 / 역질문
- **RQ1 (목적지 그릇):** 마이/공지 화면엔 검색 같은 **우측 미리보기 패널이 없습니다.** 글랜스 클릭 "미리보기(A)"를 쓰려면 → (가) 검색 우측 패널을 그 화면에도 이식 (나) **바텀시트로 통일**(이미 보험이슈/함께해요/스크립트가 바텀시트 슬롯) (다) 그냥 해당 스페이스로 점프(B)만. **어느 쪽?** (총괄팀장 추천: 1차는 (다) 점프만으로 가볍게, 미리보기(A)는 그릇 통일 후 2차)
- **RQ2 (실시간성):** `_diaCount`는 화면 진입 시 1회. 글랜스 카운트도 진입 시 1회면 충분한가, 실시간 갱신 필요?
- **RQ3 ("새 자료" 기준):** `created_at >= 오늘`인가, **마지막 방문(lastread) 이후**인가? (lastread 방식은 `os_lastread_*` localStorage가 이미 알림에서 쓰임)

### 권장 분할 (회귀 최소·에러 없이)
- **0차:** 스페이스 색상 2단계 머지·검수 완료 (선행)
- **1차:** `.glance-row` 부품 CSS + 마이 스페이스의 팀/지점 한 줄 요약(카운트 = `_diaCount` scope 확장) + **클릭=해당 스페이스 점프(B)** — 그릇 신설 없이 가벼움
- **2차:** 미리보기(A) — RQ1 그릇(바텀시트 통일 vs 우측 패널 이식) 합의 후
- **3차:** 검색 결과·공지 목록에도 동일 부품 확산
