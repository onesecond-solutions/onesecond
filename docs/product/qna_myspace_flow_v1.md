# 보험Q&A 동시-노출 흐름 — "내 책상에서 다 해결, 답만 받는다"

## 0. 한 줄 본질

> **설계사는 게시판이라는 "공간"을 찾아가지 않는다. 자기 MY SPACE("내 책상")에서 질문하고, 답만 받는다.**

질문을 하러 어딘가 게시판에 들어가는 게 아니다. 내 책상에서 질문을 던지면, 답이 알아서 내 책상으로 돌아온다.

---

## 1. 핵심 철학

- **쓰는 곳 ↔ 모이는 곳 분리** (기존 원칙): MY SPACE = 작성 / 보험 Q&A(현장의 소리) = 모임(읽기).
- **설계사 = MY SPACE 단일 허브.** 질문 작성도, 답변 수신도 MY SPACE에서. 게시판을 돌아다닐 필요 없음.
- **보험사 직원 = 보험사 자료실.** 우리 회사로 온 질문을 보고 답변.
- **현장의 소리 "질문하기" 폼 = 사용하지 않음.** (질문 입구는 MY SPACE. 2026-06-01 정정 — 현장의 소리 질문하기 실데이터화 시도(PR #239)는 이 원칙과 충돌하여 revert.)

---

## 2. 동시-노출 모델 (데이터는 "거쳐가지" 않는다)

질문·답변은 `posts`에 **한 번만 저장**된다. 여러 화면은 그 같은 데이터를 **각자 필터로 동시에** 본다. 데이터를 화면 사이로 복사·이동하지 않는다.

> 마스터 전략 §3의 **"일방향 자동 저장 = 저장소 하나, 노출 여러 곳"** 과 정합.

---

## 3. 흐름 그림

```
질문자 MY SPACE ── 질문 작성 (posts 1건 저장)
   │
   ├─ (동시) → 보험 Q&A        (모여서 보임)
   └─ (동시) → 보험사 자료실     (우리 회사 질문 = "오늘의 할 일")
                    │
                    ↓  보험사 자료실에서 답변 (posts 1건 저장, parent_post_id)
                    │
   ├─ (동시) → 보험 Q&A        (답변 노출)
   └─ (동시) → 첫 질문자 MY SPACE  (내 질문의 답 도착)
```

- **시작과 끝이 모두 질문자 MY SPACE** — 질문자는 가만히 있어도 답이 책상으로 온다.
- 가운데 **보험 Q&A = 모임**, **보험사 자료실 = 답하는 곳**.

---

## 4. 데이터 모델 (posts 한 테이블)

| 종류 | 핵심 컬럼 |
|---|---|
| **질문 row** | `question_type`(인수/상품) · `insurer_target`(보험사 범위) · `created_by`(질문자) · `branch_id` · 인수/상품 디테일(`patient_age`/`patient_gender`/`disease_name`/`diagnosis_timing`/`current_status`/`product_category`) · `status` · `title` · `content` |
| **답변 row** | `parent_post_id`(원 질문 id) · `board_type='insurer'` · 작성자=보험사 직원 |

- 위 컬럼은 **신버전(`pdnwgzneooyygfejrvbg`)에서 REST 검증 완료**(HTTP 200). **DB 신규 작업 거의 0.**
- 전용 컬럼이 없는 폼 값(직업·약복용·건강체·확인내용)은 `content`에 합쳐 저장.

---

## 5. 화면(뷰) = posts 필터

| 화면 | = posts의 필터 뷰 |
|---|---|
| **질문자 MY SPACE** | `created_by` = 나 (+ 그 질문의 답변 `parent_post_id`) |
| **보험 Q&A** | 질문·답변 모임 (노출 조건) |
| **보험사 자료실** | `insurer_target` 매칭 = 우리 회사 (+ 미답변 = "오늘의 할 일") |

→ 데이터는 한 곳, 화면은 관점만 다르다. 화면(뷰)을 하나씩 붙이면 된다.

---

## 6. "우리 회사" 매칭 규칙 (보험사 자료실)

우리 회사(예: 손해보험사)가 보는 질문 =

```
insurer_target = '전체'
OR insurer_target = '손보전체'        (우리 보종이 손해보험이면)
OR (insurer_target = '회사지정' AND 대상 = 우리 회사)
```

(생명보험사면 '손보전체' 자리에 '생보전체'.)

---

## 7. 오늘의 할 일 (보험사 자료실)

우리 회사 질문 위에서 집계한 카운트. **인수/상품 2분류 × 4수치:**

```
📋 인수 질문 : 신규 N · 미처리 N · 답변완료 N · 오늘 누적 N
💊 상품 질문 : 신규 N · 미처리 N · 답변완료 N · 오늘 누적 N
```

- 미처리/답변완료 = `parent_post_id` 답변 유무로 판별
- 신규/오늘 누적 = `created_at` 기준

---

## 8. 질문 입구 = MY SPACE

- 질문 작성 UI는 **MY SPACE**에 둔다 (현장의 소리 질문하기 폼이 아니다).
- 인수/상품 폼의 정교한 디테일(연령·성별·병력·진단시기·현재상태·상품군·보험사 범위)은 **그대로 활용** — 위치만 MY SPACE로.

---

## 9. 기존 자산과의 정합

- **master_strategy_v1 §3** — 질문 → 보험사 게시판 → 답변 → 질문자 반영(일방향 자동 저장).
- **v2_insurer_admission_phase1** — `insurer_target` 4종 + `parent_post_id` 답변 메커니즘 + 보험사 SELECT RLS 정책(§6-2) 이미 명문화.
- **organism_map_draft** — "오늘의 할 일 = 단일 엔진(롤 정의만 교체)" + "쓰는 곳 ↔ 모이는 곳".

---

## 10. 구현 단계 (데이터는 한 곳, 뷰를 하나씩)

| 단계 | 내용 |
|---|---|
| 선행 | 질문 입구를 **MY SPACE**에 (질문 작성 → `posts` INSERT) |
| 뷰 A | **보험사 자료실** = 우리 회사 질문 필터 + 오늘의 할 일 카운트 |
| 뷰 B | **답변** = 보험사 직원이 `parent_post_id`로 답변 INSERT |
| 뷰 C | **질문자 MY SPACE** = 내 질문 + 답 도착 |
| 뷰 D | **보험 Q&A** = 질문·답변 모임(읽기) |

→ 각 단계는 같은 `posts` 데이터에 **화면(뷰)을 하나 붙이는 일**. 순서는 팀장님과 정한다.

---

## 11. 회귀 신호 (이 방향에서 벗어나면 즉시 보고)

- ❌ **현장의 소리 "질문하기" 폼 부활** (질문 입구 = MY SPACE)
- ❌ 데이터를 화면 사이로 **복사·이동** (한 곳 저장 + 필터 뷰가 본질)
- ❌ 설계사가 답을 받으러 **게시판을 찾아가게** 만들기 (답은 MY SPACE로 온다)
- ❌ 인수/상품 폼 디테일을 버리고 단순화 (디테일 = 인수심사 핵심)
