## §0. 전제 — 거버넌스 규칙 (불변)
1. 채팅 스크립트 = 임태성 실장이 리딩하는 4팀 전용 팀 전략. 전 사용자용 제품 기능 아님
2. 검증 전 단계 — 없던 채팅 상담 프로세스를 만들어 "팀원 실적 향상 기여"를 검증하는
   인큐베이션 (master_strategy §5)
3. 검증 전까지 전사 절대 노출 금지. 전사 노출 여부는 검증 통과 후 별도 결재
4. 공유 기본 범위 = team이 본령. global은 현 단계 사용 안 함

## §1. 사실관계 실측 (2026-06-08)
- 단체방(team_notices) = team_id 격리 작동: scope='team_internal' AND
  team_id=my_team_id() [코드 기준: migrations/2026-05-17_branch_feed_setup.sql:69]
- 채팅 스크립트(scripts) = 격리 없음(personal/global 둘뿐) → 현재 채팅은 global 저장
- team_invitations 인프라 실재 (IS NOT NULL 가드 모범 패턴)
- team_id NULL = 전사 11건(4팀 내 1명 포함) — team 격리 시 조용한 빈 화면 위험
- 라이브 pg_policies 최종 확정 = 크롬 1회 (team_notices·scripts)
→ 결론: 팀 격리 인프라는 단체방에 검증된 형태로 존재. 채팅은 같은 패턴 이식

## §2. 데이터 설계 — team 범위 이식
- scripts에 team_id 컬럼 추가(nullable) + scope 값 'team' 허용
- RLS (team_notices 패턴 이식, SECURITY DEFINER 함수 재사용):
  SELECT: is_admin() OR personal(본인) OR global
          OR (scope='team' AND team_id IS NOT NULL AND team_id=my_team_id())
  INSERT(team): role ∈ {ga_branch_manager, ga_manager} AND team_id=my_team_id() + admin
  ※ IS NOT NULL 가드 필수 (NULL=NULL 함정 차단)
- 채팅 작성 기본값 = scope='team' + 작성자 team_id 자동 부여 (사용자 선택 불필요)
- 기존 global 채팅 시드 → team 이관 UPDATE (결재 SQL)
- 본 절 전체 = DB·RLS 변경 = SQL은 PR 파일 + 실행은 팀장님 단독

## §3. 팀 구성 프로세스 — 결정 3건
1. 팀 생성: 1단계 = 운영(admin) 세팅 (4팀 기존재) / 추후 = 지점장·실장 생성 흐름
2. 팀원 유입: 1단계 = 운영 배정 + team_id 백필 / 추후 = 가입 시 팀 선택(id 매핑,
   가입 폼 개선과 한 세트) + team_invitations 초대 흐름 재사용
3. 격리 지점: RLS가 본벽(§2), 프론트는 표시 제어만

## §4. 절대 비노출 가드 체크리스트
- [ ] RLS: scope='team' 행이 타팀·비로그인·보험사 계열·team NULL에서 0건
- [ ] 팀 채팅 조회 쿼리 = scope=team 한정 (global 조회 경로 제거)
- [ ] 채팅 작성 토글 = 매니저급에게만 노출 (공지 모델 유지)
- [ ] 실계정 3종 검증: 4팀원=보임 / 타팀·타지점=안 보임 / 보험사=안 보임 + 쿼리 우회 0
- [ ] 라벨: "공유 채팅" → "팀 채팅" 계열로 정확화

## §5. 화면 구조 개편 — 저장소/작업대 분리 (팀장님 제안, v1.1 신규)
원칙: 저장소(어디 있나)와 작업대(뭘 꺼내 쓰나)를 분리한다.

| 공간 | 역할 | 격리 |
|---|---|---|
| MY SPACE 오피스 | 내 것의 저장소 — 스크립트 칩을 [마이 스크립트]/[마이 채팅]으로 분리 | 본인 |
| 단체방 | 팀 것의 저장소 — [공유 스크립트][공유 채팅][공유 자료] 탭 신설 | team_id (기존 검증된 벽) |
| 바텀시트 | 개인 작업대 — "+추가하기"로 담은 것만 + 팀 공유분 기본 노출 | 개인 |

세부:
1. 단체방 공유 탭 = 화면 위치. 데이터는 §2의 scripts(scope='team').
   공유 자료의 데이터 위치(library 확장 vs team_notices 첨부)는 구현 시 결정·보고
2. 바텀시트 콜드스타트 방지: 실장이 공유한 팀 콘텐츠는 시트에 기본 노출
   (개인이 제거 가능) + 개인 핀(+추가)은 그 위에 누적.
   리딩 콘텐츠 자동 전달 = 검증 목적(팀원이 도입 세트를 쓰는가)과 직결
3. 핀 저장: 1단계 localStorage(기기별, DB 0) → 2단계 DB 동기화
   (알림 Phase1→2 검증된 경로 재사용)
4. 시트 칩 비대화 해소: 고정 콘텐츠 칩 나열 폐지 방향 — 시트는 복붙 트레이 본연으로

## §6. 구현 순서 (결재 후)
1. 4팀 team_id 백필 SQL (PR 파일 → 팀장님 실행)
2. scripts team 컬럼 + RLS SQL (PR 파일 → 팀장님 실행)
3. 앱: 작성 기본 scope='team' / MY SPACE 칩 분리 / 단체방 공유 탭 /
   시트 = 팀 기본 노출 + 핀(+추가, localStorage)
4. 기존 global 시드 → team 이관 SQL
5. 실계정 3종 검증(§4) → 통과 시 팀 오픈(공지 발송)
6. 검증 지표: 시트 복사 횟수 → 팀원 실적 상관 관찰 = 인큐베이션 종료 판단 근거
