## §0. 전제 — 정렬된 이해 (불변)

1. **채팅 스크립트 = 임태성 실장이 리딩하는 4팀 전용 팀 전략이다.** 전 사용자용 제품 기능이 아니다.
2. **검증 전 단계다.** 현장에 채팅 상담 프로세스는 사실상 존재하지 않으며, 이것은 없던 프로세스를 만들어 "팀원 실적 향상에 기여하는가"를 검증하는 인큐베이션이다 (master_strategy §5).
3. **검증 전까지 전사 절대 노출 금지.** 사이트 전체 노출 여부는 미정이며, 검증 통과 후 별도 결재 사항이다.
4. 따라서 공유의 기본 범위는 global이 아니라 **team이 본령**이다. global은 예외이며 현 단계에서 사용하지 않는다.

## §2. 데이터 설계 — team 범위 이식

- scripts에 `team_id` 컬럼 추가(nullable, 기존 행 영향 0) + scope 값 `team` 허용
- RLS (team_notices 패턴 이식, SECURITY DEFINER 함수 재사용):
  - SELECT: `is_admin() OR scope='personal'(본인) OR scope='global' OR (scope='team' AND team_id IS NOT NULL AND team_id = my_team_id())`
  - INSERT(team): `role ∈ {ga_branch_manager, ga_manager} AND team_id = my_team_id()` + admin
  - `IS NOT NULL` 가드 필수 (team_invitations 모범 패턴 — NULL=NULL 함정 차단)
- 채팅 작성 기본값 = **scope='team' + 작성자의 team_id 자동 부여** (사용자 선택 불필요 — "내 팀"이 기본)
- 기존에 global로 저장된 채팅 시드(도입① 등) → team으로 이관 UPDATE (결재 SQL)
- ⚠️ 본 절 전체 = DB·RLS 변경 = **SQL은 PR 파일 + 실행은 팀장님 단독** (불변 원칙)

## §3. 팀 구성 프로세스 — 결정 3건

| # | 질문 | 1단계(베타) 권고 | 정식(추후) |
|---|---|---|---|
| 1 | 팀은 누가 만드나 | **운영(admin) 세팅** — 4팀은 이미 존재 | 지점장/실장 생성 흐름 |
| 2 | 팀원은 어떻게 들어오나 | **운영 배정 + team_id 백필** (4팀 NULL 1명 포함, 전사 11건 중 4팀분 우선) | 가입 시 팀 선택(id 매핑 — 가입 폼 개선과 한 세트, 기예약) + team_invitations 초대 흐름 재사용 |
| 3 | 격리는 어디서 거나 | **RLS가 본벽**(§2) + 프론트는 표시 제어만 | 동일 |

## §4. 절대 비노출 가드 체크리스트

- [ ] RLS: scope='team' 행은 타팀·비로그인·보험사 계열·NULL team에서 0건 반환
- [ ] 시트 "팀 채팅" 칩: 조회 쿼리 자체가 scope=team 한정 (global 채팅 조회 경로 제거)
- [ ] 채팅 작성 토글: 매니저급에게만 노출 (기결재 공지 모델 유지)
- [ ] 검증(실계정 3종): ① 4팀 팀원 = 보임 ② 타팀/타지점 계정 = 안 보임 ③ 보험사 테스트 계정 = 안 보임 + ?쿼리 우회 0건
- [ ] 칩 라벨: "공유 채팅" → **"팀 채팅"** 변경 (의미 정확화)

## §5. 구현 순서 (결재 후)

1. 4팀 team_id 백필 SQL (PR 파일 → 팀장님 실행) — NULL 1명 해소
2. scripts team 컬럼 + RLS SQL (PR 파일 → 팀장님 실행)
3. 앱 수정: 작성 기본 scope='team' / 시트 칩 쿼리 team 한정 / 라벨 "팀 채팅"
4. 기존 global 시드 → team 이관 SQL
5. 실계정 3종 검증 (§4) → 통과 시 팀 오픈(공지 발송)
6. 검증 지표 가동: 시트 복사 횟수 → 팀원 실적 상관 관찰 = **인큐베이션 종료 판단 근거**
