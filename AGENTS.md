## 🚨 브랜치 · 배포 규칙 (절대 준수 — 변경 없음)

1. **main 직접 push 절대 금지.** 예외 없음.
2. **feature 브랜치에서 작업** — `feat/…` `fix/…` `docs/…` `chore/…`
3. **산출 = PR.** 자동 머지 금지. PR을 올려두고 멈춘다.
4. **머지 흐름:** PR → **총괄팀장 검수** → **대표님 승인** → **AI팀이 머지**. 코덱스는 머지하지 않는다.
5. **구현이 끝나면 무거운 검증 전에 먼저 커밋해 산출물을 보존**한다(작업 중단 시 유실 방지).
6. **한 작업 단위로 묶어 PR.** 작은 수정마다 PR 남발 금지(Netlify Preview 크레딧).

## 제품 코드 규칙

**대상:** `app.html`(단일 대형 파일) · `pages/*.html` · `js/*.js` · `css/*.css`

- **디자인 토큰만 사용, 하드코딩 금지.** v2 색상 토큰 실효 정의 = `css/app-core.css` 상단 `:root`(`--bg` `--s1` `--s2` `--bd` `--tp` `--ts` `--tf` `--ac` …). 타이포·radius 토큰도 동일.
- **직각 모서리 금지** — 최소 `--radius-sm`.
- **CSS 스코프 격리** — 화면 전용 스타일은 그 화면 id 하위로 스코프 + 네임스페이스 프리픽스(예: `#v-xfile .xf-…`). **전역 누출 0.**
- **레이아웃 폭·열 수를 임의로 고정하지 마라.** 열 수는 화면 폭이 정한다(`repeat(auto-fit, minmax(...))`). 모바일 폭(560px 등)을 데스크톱에 강제하지 말 것. *(2026-07-17 대표 지적 — 총괄이 잘못 지시해 벌어진 사고)*
- **모바일에서 안 깨지는 것이 최우선.** 고객이 폰으로 여는 화면이 있다.
- **전역 함수는 `window.*`로 등록.**
- **XSS** — `innerHTML`에 사용자 입력을 직접 넣지 말고 escape.
- **라이브 용어로 소통** — 화면·기능 이름은 실제 라이브 화면에 보이는 용어 기준. 코드 식별자(함수명·뷰 id·테이블명)는 옛 이름이 남아 있어도 그건 내부 식별자일 뿐 화면 이름이 아니다. 사전 = `docs/ops/live_term_glossary.md`.
- **식별자 리네임은 회귀 위험이 크다** — 표시 텍스트만 바꾸고 안정키(함수·뷰 id·테이블)는 유지.

## 작업 원칙

- **지시서에 확정된 사양만 구현한다.** 상상해서 기능·더미 콘텐츠를 만들지 마라. 빈 페이지를 만들라면 비워둔다.
- **기존 자산을 먼저 찾아 재사용한다.** 새로 짓기 전에 비슷한 게 있는지 확인(예: 화면 패턴·문구·공용 헬퍼). **단, 맥락 없이 베끼지 마라** — 참조 대상이 이 작업에 맞는지 판단하고, 안 맞으면 판단해서 다르게 하고 그 이유를 보고한다.
- **보험 도메인 문구를 창작하지 마라.** 기존 자산의 문구가 대표님의 영업 언어다.
- **범위를 지켜라.** 지시서가 파일을 한정하면 그 파일만. 다른 파일이 필요하면 멈추고 보고.
- **검증 후 보고.** 문법 검사(`node --check`) + 실제 화면 렌더 확인. **증거 없이 "완료"라고 쓰지 마라.** 못 한 검증은 못 했다고 쓴다.
- **막히면 추측으로 밀지 말고** 어디서 막혔는지 정확히 보고하고 멈춘다.
- **완료 보고에 포함:** 브랜치 · 커밋 해시 · PR 번호/URL · 수정 파일·라인 · 검증 결과와 근거 · **못 지킨 항목**.
